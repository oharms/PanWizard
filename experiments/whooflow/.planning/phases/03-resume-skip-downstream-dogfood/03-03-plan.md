<objective>
Add child-process tracking to `src/executor.js` so the SIGINT/SIGTERM handler in `cli.js` (wired in Plan 04) can drain in-flight children: send SIGTERM, wait briefly, persist final state, exit. The public contract of `runTask(task, opts)` does not change — same return shape, same `'close'`-event resolution discipline. The only additions are: (1) a module-level `activeChildren` Set that records every live spawn, (2) `getActiveChildren()` and `killActiveChildren(signal)` exports for the CLI to call, (3) `opts.signal` handling so an AbortSignal abort kills the live child mid-run (the existing `retry.js` scaffold's signal becomes meaningful for spawned children, not just sleeping backoffs).

Purpose: DOG-03 (kill-and-resume) requires that a SIGINT during a running task drains the child via SIGTERM-then-SIGKILL, persists final state, and exits with the right code. The drain logic itself lives in `cli.js` (Plan 04 wires it into the existing single-handler-at-startup), but it needs a way to enumerate and kill in-flight children. Tracking the children in the executor (where they're spawned) is the right separation: `cli.js` doesn't need to know about subprocess IDs or how shells are spawned; it just calls `killActiveChildren('SIGTERM')` and waits.

This plan ships the infrastructure ONLY. The actual drain protocol (SIGTERM → 5s wait → SIGKILL → persist → exit) is wired in Plan 04 inside the SIGINT handler. The drain logic depends on this plan + Plan 01's mergeState (for resume after kill) + Plan 02's propagateSkip (irrelevant on kill, but already shipped). By keeping this plan scoped to executor-only, Wave 1 plans 01/02/03 stay file-disjoint and run in parallel.

Output: An updated `src/executor.js` with active-children tracking and the two new exports, plus 3+ new tests in `test/executor.test.js` verifying tracking behavior + abort-signal kill semantics. No changes to `cli.js` (Plan 04) or `scheduler.js` (Plan 04).
</objective>

Modify `src/executor.js` to track each spawned `ChildProcess` in a module-level Set. The set is populated immediately after `spawn()` returns and drained inside the same `'close'` / `'error'` handlers that resolve the Promise. Add `getActiveChildren()` and `killActiveChildren(signal)` exports. Add optional `opts.signal` handling to `runTask` so an aborted signal kills the live child.

The full updated file:

```js
// Phase 1 Plan 03 — child_process spawn + 'close' discipline.
// EXEC-01 + EXEC-02 + pitfall C3 + pitfall C6:
//   - shell: true for cross-platform cmd execution (cmd.exe on Windows, /bin/sh on POSIX)
//   - resolve on 'close' (NOT 'exit') — 'close' fires after stdio streams drain.
//   - null exit code (signal-killed) is treated as failure with exit_code = -1.
//   - windowsHide: true suppresses the conhost flash on Windows.
//   - stdio: 'inherit' so child output streams live to the user's terminal.
//
// Phase 3 Plan 03 — active-children tracking for SIGINT drain protocol (DOG-03).
//   - Module-level activeChildren Set: every spawn() result is added immediately and
//     removed inside the 'close' handler (the same callback that resolves the Promise).
//   - getActiveChildren() returns a snapshot for the CLI's drain handler (Plan 04).
//   - killActiveChildren(signal) iterates and calls child.kill(signal) — best-effort.
//   - opts.signal (AbortSignal) — when aborted mid-run, kills the live spawned child
//     and removes it from the active set; the 'close' handler then resolves with
//     ok:false / exit_code:-1 / signal:'SIGTERM' (or whatever the child reports).

import { spawn } from 'node:child_process';

const activeChildren = new Set();

/**
 * Snapshot of currently-running spawned children. Used by the SIGINT handler
 * in cli.js (Plan 04) to send SIGTERM during graceful shutdown.
 */
export function getActiveChildren() {
  return [...activeChildren];
}

/**
 * Best-effort signal delivery to every active child. Called by the SIGINT
 * handler in cli.js (Plan 04). A child that has already exited but not yet been
 * removed from the set is harmless — child.kill() on a dead pid is a no-op.
 */
export function killActiveChildren(signal = 'SIGTERM') {
  for (const child of activeChildren) {
    try {
      child.kill(signal);
    } catch {
      // already-exited / EPERM — best effort
    }
  }
}

/**
 * Phase 1 has NO retry loop — that's Phase 2 (retry.js). runTask runs exactly once.
 * Returns Promise<{ok, exit_code, signal, error?}>. Always resolves (never rejects).
 *
 * @param {Object} task - normalized task with { id, cmd }
 * @param {Object} [opts]
 * @param {AbortSignal} [opts.signal] - if aborted mid-run, kills the spawned child.
 *                                       The 'close' handler still resolves normally.
 */
export function runTask(task, opts = {}) {
  return new Promise((resolve) => {
    let resolved = false;
    let spawnErr = null;

    const child = spawn(task.cmd, {
      shell: true,
      stdio: 'inherit',
      windowsHide: true,
    });

    activeChildren.add(child);

    // P3: opts.signal (AbortSignal) — when aborted mid-run, kill the live child.
    // The 'close' handler still resolves with the eventual exit/signal info.
    let abortListener = null;
    if (opts.signal) {
      if (opts.signal.aborted) {
        // Already aborted — kill immediately. The 'close' handler will resolve.
        try { child.kill('SIGTERM'); } catch { /* no-op on dead pid */ }
      } else {
        abortListener = () => {
          try { child.kill('SIGTERM'); } catch { /* no-op */ }
        };
        opts.signal.addEventListener('abort', abortListener, { once: true });
      }
    }

    function cleanupListeners() {
      if (abortListener && opts.signal) {
        opts.signal.removeEventListener('abort', abortListener);
      }
    }

    child.on('error', (err) => {
      // Spawn-time error (e.g., ENOENT for the shell). Defer to 'close' if it follows;
      // otherwise fall back here.
      spawnErr = err;
      if (!resolved) {
        setImmediate(() => {
          if (resolved) return;
          resolved = true;
          activeChildren.delete(child);
          cleanupListeners();
          resolve({ ok: false, exit_code: -1, signal: null, error: err.message });
        });
      }
    });

    child.on('close', (code, signal) => {
      if (resolved) return;
      resolved = true;
      activeChildren.delete(child);
      cleanupListeners();
      if (spawnErr) {
        resolve({ ok: false, exit_code: -1, signal: null, error: spawnErr.message });
        return;
      }
      if (code === null) {
        // Signal-killed (SIGTERM, SIGKILL, OOM). Treat as failure per EXEC-02.
        resolve({ ok: false, exit_code: -1, signal: signal ?? null });
        return;
      }
      resolve({ ok: code === 0, exit_code: code, signal: null });
    });
  });
}
```

**DO NOT** change the resolution discipline: still resolve on `'close'`, NOT `'exit'`. Pitfall C3 is structural — `'exit'` fires before stdio drains and would truncate output.

**DO NOT** make `runTask` reject on abort. The Promise still resolves (with `ok: false`) — the existing scheduler code in `runTaskWithRetry.then(...)` doesn't have a `.catch` for kill scenarios; making runTask reject would be a breaking change to the contract that retry.js depends on.

**DO NOT** export the `activeChildren` Set directly. It's a module-private detail; the CLI accesses it only via `getActiveChildren()` and `killActiveChildren()`.

**DO NOT** add a global `setMaxListeners` call. AbortSignal listener leak prevention is per-spawn: every `addEventListener('abort', ...)` is paired with a `removeEventListener` in cleanupListeners(), and the signal's listener count never exceeds the active-children count (bounded by `--concurrency`).

**DO NOT** change `retry.js`. Plan 04 will wire opts.signal through from cli.js → runFlow → runTaskWithRetry → runTask. This plan only readies the executor.

```js
// (Append to existing test/executor.test.js — keep the existing import block and tests.)

import { getActiveChildren, killActiveChildren } from '../src/executor.js';

test('Phase 3: getActiveChildren reflects in-flight children, drains on close', async () => {
  // A short-lived task. Before it resolves, the active set should contain the child.
  const taskPromise = runTask({ id: 't', cmd: 'node -e "setTimeout(()=>{}, 100)"' });
  // Synchronously after kicking off runTask, the spawn is queued — give one tick for
  // spawn() to return and the .add(child) line to run.
  await new Promise((r) => setImmediate(r));
  const inFlight = getActiveChildren();
  assert.equal(inFlight.length, 1, 'one child should be in-flight');

  await taskPromise;
  // After resolution, the active set should be empty.
  assert.equal(getActiveChildren().length, 0, 'active set drained on close');
});

test('Phase 3: killActiveChildren("SIGTERM") kills in-flight children — runTask resolves with ok:false', async () => {
  // Long-sleeping task. We kill it via the public API and assert runTask reports failure.
  const taskPromise = runTask({ id: 'long', cmd: 'node -e "setTimeout(()=>{}, 30000)"' });
  await new Promise((r) => setImmediate(r));
  killActiveChildren('SIGTERM');
  const result = await taskPromise;
  assert.equal(result.ok, false, 'kill produces failure');
  assert.equal(result.exit_code, -1, 'signal-killed → exit_code -1 per EXEC-02');
  // signal field may be 'SIGTERM' on POSIX or null on Windows (where SIGTERM maps to TerminateProcess
  // and the child reports null signal). Either is acceptable.
  assert.equal(getActiveChildren().length, 0, 'active set drained after kill');
});

test('Phase 3: opts.signal abort mid-run kills the child', async () => {
  const ac = new AbortController();
  const taskPromise = runTask(
    { id: 'abortable', cmd: 'node -e "setTimeout(()=>{}, 30000)"' },
    { signal: ac.signal },
  );
  await new Promise((r) => setImmediate(r));
  ac.abort();
  const result = await taskPromise;
  assert.equal(result.ok, false, 'abort produces failure');
  assert.equal(result.exit_code, -1);
  assert.equal(getActiveChildren().length, 0);
});

test('Phase 3: opts.signal already aborted at runTask call → child killed before close', async () => {
  const ac = new AbortController();
  ac.abort();
  const result = await runTask(
    { id: 'preaborted', cmd: 'node -e "setTimeout(()=>{}, 30000)"' },
    { signal: ac.signal },
  );
  assert.equal(result.ok, false);
  assert.equal(result.exit_code, -1);
});
```

Note on cross-platform behavior: on Windows, `child.kill('SIGTERM')` translates to `TerminateProcess` (immediate). On POSIX, the child receives SIGTERM and may or may not handle it before exiting. Either way, the `'close'` event fires and the runTask Promise resolves. The tests above don't assert on the specific `result.signal` value because Windows reports null where POSIX reports 'SIGTERM' — both are valid kill outcomes.

**DO NOT** sleep or use polling intervals to wait for the child to exit. Awaiting `taskPromise` is the correct synchronization primitive — the Promise resolves when `'close'` fires.

**DO NOT** add per-test timeouts beyond `node:test`'s default. The killed children resolve in well under 1 second; the default test timeout (in seconds) is plenty.
