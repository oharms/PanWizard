### Open questions for verifier

- **Q-1:** The Wave 0 probe in `encode()` checks if `encoding_rs::UTF_16LE.encode(text).len() == text.encode_utf16().count() * 2`. This test passed (raw bytes emitted). However, `encoding_rs`'s `Encoding::encode()` for UTF-16 LE/BE technically goes UTF-8 → UTF-8 (the encoding_rs docs say UTF-16 LE/BE encode() returns UTF-8 for the Web Encoding spec). The tests show the current behavior is correct raw bytes — but if a future encoding_rs version changes this behavior, the probe + fallback in `src/encoding.rs::encode` will transparently switch to the manual encoder. Worth re-running the Wave 0 test after any encoding_rs version bump.

## Wave 0 Spike Resolution

**encoding_rs::UTF_16LE.encode("hello") emits raw UTF-16 LE bytes: YES**

- Test: `encode(DetectedEncoding::Utf16Le, "hello")` returns 12 bytes: `[0xFF, 0xFE, 0x68, 0x00, 0x65, 0x00, 0x6C, 0x00, 0x6C, 0x00, 0x6F, 0x00]`
- The probe in `src/encoding.rs::encode` (`rs_bytes.len() == text.encode_utf16().count() * 2`) evaluates `true`
- Therefore: `encoding_rs::UTF_16LE.encode()` is used directly; the `encode_utf16_le_manual` fallback is compiled but not invoked
- Resolution: **Document `encoding_rs = "0.8.35"` as confirmed-working for UTF-16 LE/BE encode. Re-verify on crate version bumps.**
