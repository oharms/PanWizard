## Plan Decisions

(See @./.claude/pan-wizard-core/references/handoff-decisions.md for the schema.)

### Locked (executor MUST follow)

- **D-1**: The size assertion runs against the actual release binary at
  `target/release/notepadrs.exe` (or `target/x86_64-pc-windows-msvc/release/notepadrs.exe` if
  the cross-target path is used). The test PRE-CONDITION is that the binary exists; if it
  doesn't, the test panics with a clear "run `cargo build --release` first" message rather
  than silently passing. Why: a missing binary is not a passing gate. Source: research.md
  §"Common Pitfalls" Pitfall 6.
- **D-2**: 10MB threshold = `10 * 1024 * 1024 = 10_485_760` bytes. NOT `10_000_000`. Why:
  matches the SC-11 spec language "<10MB" with the binary-size convention. Source:
  requirements.md BUILD-02.
- **D-3**: The test-count assertion is enforced by counting "test result: ok. {N} passed"
  lines from `cargo test` output, OR by hard-coding a known-good list of test files / counts.
  Recommended approach: hard-code an inventory check in tests/ship_gate.rs that asserts each
  expected test file is present (file existence + non-empty). This is more robust than parsing
  test output and runs in any CI shape. Why: cargo test counts are environment-dependent (some
  tests may be `#[ignore]`-gated by feature flags); a file-inventory check is stable.
- **D-4**: Test file name is `tests/ship_gate.rs` (NOT `tests/release_dispatch.rs` or
  `tests/setup_check.rs`). Why: AppCompat shim error 740 forbids dispatch/install/update/setup.
- **D-5**: ship-checklist.md is a written procedure for the v1.0.0 release; it does NOT auto-tag
  or auto-push to a remote. Tagging and pushing are explicit human actions performed AFTER the
  checklist's manual sign-off. Why: irreversible actions (git tag + push) should not be in
  Claude's autonomous path.
- **D-6**: This plan does NOT depend on Plan 05-04 (`autonomous: true`, parallel with 05-04
  in Wave 3). Why: ship gate is automated assertion of facts about the artifacts (size, test
  count), independent of the manual dogfood verification. They can run concurrently. Source:
  research.md §"Plan Sizing" Wave 3.

### Open (executor's discretion within constraints)

- **O-1**: Whether `tests/ship_gate.rs` actually invokes `cargo build --release` from within
  the test (heavy; potentially slow), or REQUIRES the binary be already built (lightweight;
  but introduces a documented precondition). Recommend: lightweight — test asserts the binary
  exists and is sized correctly, with a clear panic message if not. CI runners build then test
  separately. Constraint: test must be reproducible.
- **O-2**: The test-count gate methodology. Three options:
  - (a) File-inventory check (per D-3 recommendation) — robust, stable
  - (b) Spawn `cargo test --release` and parse output — heavy, environment-dependent
  - (c) Hard-coded list of expected `#[test]` functions with `include_str!` parsing — fragile
  Pick (a). Constraint: must fail loudly if a test file is missing.
- **O-3**: Whether to also assert binary properties beyond size — DLL dependencies (BUILD-03),
  MSRV (BUILD-05), etc. Recommend: NO. Those are best verified by CI shell scripts, not Rust
  test code. ship-checklist.md should mention them as manual sign-off items.

### Considered and rejected

- **R-1**: Run the size+count gates as a `build.rs` post-build hook. Rejected: build.rs runs
  before code compiles; it can't see the final exe size. Tests are the right home.
- **R-2**: Create a separate `[[bin]] tools/check-binary-size.rs` instead of a test. Rejected:
  developers run `cargo test`, not arbitrary tools. Failing the test integrates with CI
  naturally.
- **R-3**: Auto-tag and auto-push to remote on green ship gate. Rejected by D-5 — release
  signing/tagging is the human's call.
- **R-4**: Parse `cargo test` output to count tests. Rejected by D-3 / O-2 — environment-
  dependent and fragile.

1. **tests/ship_gate.rs** — Create:

   ```rust
   //! Plan 05-05: SC-11 / SC-12 ship gate.
   //!
   //! These tests fail the build if the v1.0.0 release thresholds are breached.
   //! - SC-11 / BUILD-02: release binary ≤ 10MB
   //! - SC-12 / TEST-01: ≥ 30 cumulative passing tests across the test suite
   //!
   //! The size gate REQUIRES the release binary to be built first — run
   //! `cargo build --release` before `cargo test`. CI runs build then test in sequence.

   use std::path::PathBuf;

   const TEN_MEGABYTES: u64 = 10 * 1024 * 1024; // SC-11 ceiling

   /// Locate the release binary. Tries the default target/release path first, then the
   /// cross-target path used in some CI configs.
   fn release_binary() -> Option<PathBuf> {
       let candidates = [
           PathBuf::from("target/release/notepadrs.exe"),
           PathBuf::from("target/x86_64-pc-windows-msvc/release/notepadrs.exe"),
       ];
       candidates.into_iter().find(|p| p.is_file())
   }

   #[test]
   fn binary_size_under_10mb() {
       let path = match release_binary() {
           Some(p) => p,
           None => panic!(
               "Release binary not found. Run `cargo build --release` before `cargo test`. \
                Looked at: target/release/notepadrs.exe and \
                target/x86_64-pc-windows-msvc/release/notepadrs.exe"
           ),
       };
       let size = std::fs::metadata(&path)
           .unwrap_or_else(|e| panic!("metadata({}): {e}", path.display()))
           .len();
       assert!(
           size <= TEN_MEGABYTES,
           "SC-11 / BUILD-02 BREACHED: {} is {} bytes (>{}MB ceiling). \
            Investigate dependency additions or LTO regression.",
           path.display(),
           size,
           TEN_MEGABYTES / (1024 * 1024)
       );
       eprintln!("ship_gate: binary size = {size} bytes ({:.2} MB) — OK", size as f64 / (1024.0 * 1024.0));
   }

   #[test]
   fn test_inventory_meets_minimum_test_count() {
       // SC-12 / TEST-01: ≥ 30 cumulative passing tests.
       // We assert by file inventory: every required test file exists and is non-empty.
       // Each file contributes its known cargo-test count; the running suite proves them passing.
       let required_test_files: &[(&str, usize)] = &[
           // (path, minimum_test_count_in_this_file)
           // Phase 1
           ("tests/cargo_profile.rs",          1),
           ("tests/com_init.rs",               2),
           ("tests/panic_safety.rs",           6),
           ("tests/roundtrip_utf8_ascii.rs",   3),
           ("tests/undo_property.rs",          6),
           // Phase 2
           ("tests/encoding_cascade.rs",       16),
           ("tests/eol_detection.rs",          14),
           ("tests/menu_action_tests.rs",      10),
           ("tests/open_errors.rs",            11),
           ("tests/phase02_coverage.rs",       26),
           ("tests/roundtrip_matrix.rs",       14),
           // Phase 3
           ("tests/find_engine.rs",            31),
           ("tests/find_replace_integration.rs", 8),
           ("tests/find_replace_pure.rs",      8),
           ("tests/find_worker.rs",            12),
           ("tests/phase03_additional.rs",     35),
           ("tests/replace_engine.rs",         8),
           ("tests/tab_model.rs",              18),
           // Phase 4
           ("tests/gutter_layout.rs",          15),
           // highlight_perf is #[ignore] release-only; skip from inventory
           ("tests/phase04_data_shape.rs",     13),
           ("tests/status_bar_labels.rs",      19),
           ("tests/syntax_tokenizers.rs",      39),
           // Phase 5
           ("tests/word_wrap.rs",              4),  // Plan 05-01
           ("tests/recent_files.rs",           10), // Plan 05-02
           ("tests/drag_drop_pure.rs",         6),  // Plan 05-03
           // tests/dogfood_marker.rs is OPTIONAL (Plan 05-04 O-1) — not in this list
       ];

       let mut total = 0usize;
       let mut missing: Vec<&str> = Vec::new();
       for (path, min_count) in required_test_files {
           if !std::path::Path::new(path).is_file() {
               missing.push(path);
           } else {
               total += min_count;
           }
       }

       assert!(
           missing.is_empty(),
           "ship_gate: missing required test files: {missing:?}. \
            Add or repair before shipping v1.0.0."
       );
       assert!(
           total >= 30,
           "SC-12 / TEST-01 BREACHED: inventory minimum is {total} tests, \
            but SC-12 requires ≥30."
       );
       eprintln!("ship_gate: inventory minimum = {total} tests across {} files — OK",
           required_test_files.len());
   }
   ```

   Adjust the per-file minimum counts to match the actual numbers from each summary file. The
   numbers above are taken from research.md §"Current Test Inventory" — but executor MUST grep
   each test file and verify the number is current (test counts may have shifted in plans
   05-01..03). If any number is OUTDATED, lower it to a stable minimum (e.g., the lowest count
   ever observed for that file) — the gate is "≥ X", so undershooting in the inventory is safe.
