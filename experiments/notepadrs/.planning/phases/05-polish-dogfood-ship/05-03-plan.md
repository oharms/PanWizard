<objective>
Implement WM_DROPFILES drag-and-drop file opening (DND-01). Drop a file (or multiple files)
onto the notepadrs window and they open in new tabs.

Purpose: Closes SC-10. WM_DROPFILES is the simple Win32-shell API (~10 LOC) — chosen over
IDropTarget COM (~150 LOC of vtable scaffolding) per the locked decision in stack.md
(lines 230-249) and research.md §"Alternatives Considered". OleInitialize was already paired
with OleUninitialize in main.rs:68/73 (Phase 1), so the substructure is in place.

Output: Two modifications to src/dispatch.rs (DragAcceptFiles in WM_CREATE; WM_DROPFILES handler
arm in WndProc), and a small pure-logic test file for the multi-file path-buffer parsing.

**Depends on Plan 05-02:** Recent Files persistence is wired into `open_path_external` by
05-02. Drag-drop reuses that same code path, so dropped files automatically appear in Recent
Files. This is why 05-03 is in Wave 2 (after 05-02), not Wave 1.
</objective>

## Plan Decisions

(See @./.claude/pan-wizard-core/references/handoff-decisions.md for the schema.)

### Locked (executor MUST follow)

- **D-1**: Use `WM_DROPFILES` + `DragAcceptFiles` + `DragQueryFileW` + `DragFinish`. NOT
  `IDropTarget` / `RegisterDragDrop`. Why: SC-10 only requires file-path drops; WM_DROPFILES
  is the documented standard for that case. Source: stack.md lines 230-249 (locked decision)
  + research.md §"Alternatives Considered".
- **D-2**: `DragAcceptFiles(hwnd, TRUE)` is called EXACTLY ONCE in `WM_CREATE`, AFTER the
  window is fully constructed (status bar + tab strip + first tab pushed). Why: idempotent but
  one syscall is enough; OleInitialize must be in effect (it is — main.rs:68 calls it before
  CreateWindowExW). Source: research.md §"Pattern 3" + §"Anti-Patterns".
- **D-3**: WM_DROPFILES handler iterates with `DragQueryFileW(hdrop, idx, ...)` from idx=0
  to count-1, where count = `DragQueryFileW(hdrop, 0xFFFFFFFF, None)`. Why: documented Win32
  pattern. Source: research.md §"Code Examples — Example 3".
- **D-4**: `DragFinish(hdrop)` is called AT THE END of the WM_DROPFILES handler, on EVERY
  exit path (including continue/early-break). Why: HDROP holds OS resources; not finishing
  leaks until process exit. Source: research.md §"Anti-Patterns" line "Forgetting DragFinish".
- **D-5**: Multi-file drop sequencing matches Plan 03-02 D-14 fresh-tab heuristic: the FIRST
  dropped file uses the fresh-tab heuristic (load into active tab if it's a fresh empty
  Untitled); SUBSEQUENT drops always push new tabs. Why: avoids creating intermediate empty
  tabs (research.md §"Common Pitfalls" Pitfall 5). Source: research.md §"Common Pitfalls"
  Pitfall 5.
- **D-6**: Folder drops are silently ignored via `path.is_file()` filter. Why: SC-10 says
  "file" not "folder"; opening a directory is undefined; folder-iteration is v2 work. Source:
  research.md §"Open Questions" #4.
- **D-7**: Dropped paths route through `open_path_external` — the SAME function used by
  CLI-arg open, do_file_open dialog, and Plan 05-02's Recent click handler. Why: single source
  of truth for "successful path open"; transitively gives drag-drop the Recent Files push
  behavior wired by 05-02 D-10.
- **D-8**: Tests in `tests/drag_drop_pure.rs` are pure-logic only (no Win32 / no HDROP).
  Test the path-list parsing helper that converts raw UTF-16 buffers to `Vec<PathBuf>`, plus
  the folder-vs-file filter logic. Why: WM_DROPFILES handler in dispatch.rs is the Win32 shim;
  pure helpers in `crate::dispatch_pure` (or a new helper module) carry the logic. Test file
  name avoids `dispatch`/`install`/`update`/`setup` substrings (Pitfall: AppCompat error 740).
- **D-9**: Non-UTF-8 paths (extremely rare on modern Windows) are skipped with a log line.
  Why: consistent with Plan 05-02 D-3 — String boundary semantics.

### Open (executor's discretion within constraints)

- **O-1**: Where to put the pure-logic helper. Two viable spots: (a) extend `dispatch_pure.rs`
  with a `parse_dropped_paths(buf16: &[u16]) -> Vec<PathBuf>` helper or (b) inline in
  `dispatch.rs` (no pure-logic split). Recommendation: (a) — matches the project's existing
  pure/effects split. Constraint: tests/drag_drop_pure.rs must be able to import and exercise
  the helper without a real HDROP.
- **O-2**: Whether to suppress drag-drop while a dirty tab close prompt is open (defensive
  against re-entrancy). Default: don't add explicit suppression — Win32 already serializes
  WM_DROPFILES with WM_COMMAND so the user won't be in two states at once.
- **O-3**: Whether to scroll/raise focus to the newly-opened tab on multi-file drop. Recommend:
  yes for the LAST file (so the user sees what they just dropped). Constraint: must not
  break test assertions.

### Considered and rejected

- **R-1**: `IDropTarget` / `RegisterDragDrop`. Rejected: 10× more code; not needed for file-path
  drops. Locked in stack.md.
- **R-2**: Disable RichEdit's built-in OLE drag-drop. Rejected: RichEdit 4.1 doesn't consume
  file drops by default — file drops route to parent's WM_DROPFILES; text drops within editor
  are a desirable bonus. Source: research.md §"Pattern 3 Caveat".
- **R-3**: Recursively iterate dropped folders. Rejected: SC-10 says "file"; v2 work. (D-6.)
- **R-4**: Make drag-drop accept arbitrary path strings (URLs, etc.). Rejected: WM_DROPFILES
  by definition delivers filesystem paths only.
- **R-5**: Pre-plan an authentication gate. Rejected: no auth involved.

```rust
pub const WM_DROPFILES: u32 = 0x0233;

// Functions in windows::Win32::UI::Shell:
pub unsafe fn DragAcceptFiles(hwnd: HWND, faccept: BOOL);
```

   ```rust
   /// Plan 05-03 D-8: pure-logic helper for tests. Given a raw UTF-16 buffer that
   /// represents a single path (no embedded nulls — the real WM_DROPFILES path is
   /// fetched one-at-a-time via DragQueryFileW(hdrop, idx, ...) so this helper just
   /// covers the UTF-16 → PathBuf conversion that the production handler does inline).
   ///
   /// Returns None if the buffer is empty or contains non-UTF-8-representable
   /// characters that cause the resulting PathBuf to fail .to_str() — those are
   /// skipped at the boundary per D-9.
   pub fn utf16_to_pathbuf(buf: &[u16]) -> Option<std::path::PathBuf> {
       if buf.is_empty() { return None; }
       let trimmed: Vec<u16> = buf.iter().copied().take_while(|c| *c != 0).collect();
       if trimmed.is_empty() { return None; }
       let s = String::from_utf16_lossy(&trimmed);
       Some(std::path::PathBuf::from(s))
   }

   /// Plan 05-03 D-6: drop folders, keep files. Stale (non-existent) paths are also
   /// dropped by this filter — caller should NOT pre-validate; callers depend on
   /// is_file() returning false for both directories and missing files.
   pub fn is_droppable_file(path: &std::path::Path) -> bool {
       path.is_file()
   }
   ```

3. **src/dispatch.rs** — Add the WM_DROPFILES match arm. Place it AFTER WM_INITMENUPOPUP and
   BEFORE WM_COMMAND in the dispatch switch (alphabetical-ish; near other shell messages):

   ```rust
   WM_DROPFILES => {
       use windows::Win32::UI::Shell::{DragAcceptFiles as _, DragFinish, DragQueryFileW, HDROP};
       let hdrop = HDROP(wparam.0 as *mut std::ffi::c_void);

       // 1. Probe count.
       let count = DragQueryFileW(hdrop, 0xFFFF_FFFF, None);

       // 2. For each dropped file, fetch path then route through open_path_external.
       //    First drop uses fresh-tab heuristic (mirror do_file_open D-14); subsequent
       //    drops always push new tabs.
       let mut first_drop_consumed_active = false;
       for idx in 0..count {
           // Probe length first (returns chars excluding null).
           let len_needed = DragQueryFileW(hdrop, idx, None) as usize;
           if len_needed == 0 { continue; }
           let mut buf: Vec<u16> = vec![0; len_needed + 1];
           let len = DragQueryFileW(hdrop, idx, Some(&mut buf));
           if len == 0 { continue; }

           let path = match crate::dispatch_pure::utf16_to_pathbuf(&buf[..len as usize]) {
               Some(p) => p,
               None => {
                   oslog(&format!("drop: skipped non-UTF-16 path at idx {idx}"));
                   continue;
               }
           };

           // D-6: folder filter (also catches stale/missing paths).
           if !crate::dispatch_pure::is_droppable_file(&path) {
               oslog(&format!("drop: skipped non-file path: {}", path.display()));
               continue;
           }

           // D-5: fresh-tab heuristic for the FIRST file in the drop set; new tabs after that.
           if idx == 0 && !first_drop_consumed_active {
               let active_is_fresh = app.tabs.get(app.active_tab)
                   .map(|t| t.editor.current_path.is_none()
                       && !t.dirty
                       && t.editor.original_bytes.is_empty())
                   .unwrap_or(true);
               if !active_is_fresh {
                   if let Ok(_) = app.push_empty_tab(hwnd, false) {
                       let new_idx = (app.tabs.len() - 1) as i32;
                       if let Some(t) = app.tabs.last() {
                           let _ = insert_tab_item(app.htabs, new_idx, t);
                       }
                       let _ = SendMessageW(app.htabs, TCM_SETCURSEL,
                           Some(WPARAM(new_idx as usize)), Some(LPARAM(0)));
                       switch_active_tab(app, hwnd, new_idx as usize);
                   }
               }
               first_drop_consumed_active = true;
               open_path_external(hwnd, &path);
           } else {
               // 2nd, 3rd, ... drops always push new tabs.
               if let Ok(_) = app.push_empty_tab(hwnd, false) {
                   let new_idx = (app.tabs.len() - 1) as i32;
                   if let Some(t) = app.tabs.last() {
                       let _ = insert_tab_item(app.htabs, new_idx, t);
                   }
                   let _ = SendMessageW(app.htabs, TCM_SETCURSEL,
                       Some(WPARAM(new_idx as usize)), Some(LPARAM(0)));
                   switch_active_tab(app, hwnd, new_idx as usize);
                   open_path_external(hwnd, &path);
               }
           }
       }

       // 3. ALWAYS finish — D-4. Even if the for-loop above bailed early, we still
       //    return through this path because Rust's flow control doesn't bypass
       //    statements after the loop.
       DragFinish(hdrop);
       return LRESULT(0);
   }
   ```

   ```rust
   //! DND-01 (Plan 05-03) — pure-logic tests for path parsing and folder filtering.
   //!
   //! No Win32 calls; the actual WM_DROPFILES handler in dispatch.rs is exercised
   //! manually via the dogfood marker (Plan 05-04). These tests cover the helpers
   //! that handler calls into.

   use notepadrs::dispatch_pure::{utf16_to_pathbuf, is_droppable_file};

   fn utf16_of(s: &str) -> Vec<u16> {
       s.encode_utf16().chain(std::iter::once(0u16)).collect()
   }

   #[test]
   fn parse_simple_ascii_path() {
       let buf = utf16_of(r"C:\Users\test\file.txt");
       let p = utf16_to_pathbuf(&buf);
       assert!(p.is_some());
       assert_eq!(p.unwrap().to_string_lossy(), r"C:\Users\test\file.txt");
   }

   #[test]
   fn parse_unicode_path() {
       let buf = utf16_of(r"C:\Users\café\日本語.txt");
       let p = utf16_to_pathbuf(&buf);
       assert!(p.is_some());
       let s = p.unwrap().to_string_lossy().into_owned();
       assert!(s.contains("café") || s.contains("caf\u{e9}"));
       assert!(s.contains("日本語"));
   }

   #[test]
   fn parse_empty_buffer_returns_none() {
       let buf: Vec<u16> = vec![];
       assert!(utf16_to_pathbuf(&buf).is_none());
   }

   #[test]
   fn parse_only_null_returns_none() {
       let buf: Vec<u16> = vec![0];
       assert!(utf16_to_pathbuf(&buf).is_none());
   }

   #[test]
   fn parse_handles_no_trailing_null() {
       // DragQueryFileW reports len excluding null; we pass &buf[..len], which
       // already excludes the null. Helper should still work.
       let s: Vec<u16> = "hello.txt".encode_utf16().collect();
       let p = utf16_to_pathbuf(&s);
       assert!(p.is_some());
       assert_eq!(p.unwrap().to_string_lossy(), "hello.txt");
   }

   #[test]
   fn folder_filter_rejects_directory() {
       let temp_dir = std::env::temp_dir();
       assert!(temp_dir.is_dir());
       assert!(!is_droppable_file(&temp_dir),
           "directory should not be droppable");
   }

   #[test]
   fn folder_filter_rejects_missing_file() {
       let mut path = std::env::temp_dir();
       path.push(format!("notepadrs_nonexistent_{}.txt", std::process::id()));
       let _ = std::fs::remove_file(&path);
       assert!(!is_droppable_file(&path),
           "missing file should not be droppable");
   }

   #[test]
   fn folder_filter_accepts_existing_file() {
       let mut path = std::env::temp_dir();
       path.push(format!("notepadrs_drop_test_{}.txt", std::process::id()));
       std::fs::write(&path, b"test contents").expect("write");
       assert!(is_droppable_file(&path), "regular file should be droppable");
       let _ = std::fs::remove_file(&path);
   }
   ```
