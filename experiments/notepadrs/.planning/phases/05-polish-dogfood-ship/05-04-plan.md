<objective>
Verify Phase 5 (and the v1 release as a whole) against the dogfood marker — the existential
test of notepadrs: "the developer can use notepadrs.exe to open, edit, and save the project's
own Cargo.toml without going back to another editor."

Purpose: This is the integration verification of all four Phase 5 features (word wrap, recent
files, drag-drop) plus the entire Phase 1-4 stack (multi-tab, encoding cascade, find/replace,
gutter, status bar, syntax highlighting). It's not new feature work — it's the gate that
proves the editor is ready to ship.

Output: A 9-step manual checklist document at .planning/phases/05-polish-dogfood-ship/dogfood-checklist.md
with pass/fail annotations after each step, plus an OPTIONAL automated smoke test
(tests/dogfood_marker.rs) that exercises a byte-exact Cargo.toml round-trip without
relying on GUI interaction.

This plan has a `checkpoint:human-verify` task because steps 4-9 of the checklist require
interactive Windows desktop testing that Claude cannot automate.
</objective>

## Plan Decisions

(See @./.claude/pan-wizard-core/references/handoff-decisions.md for the schema.)

### Locked (executor MUST follow)

- **D-1**: The 9-step procedure from research.md §"Dogfood Marker (operational definition)"
  MUST be copied verbatim into dogfood-checklist.md with each step extended by explicit
  pass/fail criteria. NOT paraphrased — the steps are spec; this plan only adds the verification
  scaffolding. Source: research.md §"Dogfood Marker".
- **D-2**: Step 7 (`cargo build` after the round-trip) is THE existential gate. If it fails,
  encoding or EOL was corrupted on save and v1 must NOT ship. The checklist MUST flag step 7
  as **BLOCKING — release-blocking failure**. Source: research.md §"Dogfood Marker" line "Test 7
  is the dogfood gate".
- **D-3**: Optional automated smoke test (tests/dogfood_marker.rs) is a NICE-TO-HAVE — not
  a substitute for the manual checklist. It covers byte-exact round-trip ONLY (Phase 2 path
  that was already test-verified in tests/roundtrip_matrix.rs); the GUI parts (recent files,
  drag-drop, word-wrap) cannot be automated without UI driver tooling. Source: research.md
  §"Open Questions" #5.
- **D-4**: This plan has `autonomous: false` because the checkpoint:human-verify task pauses
  for the user to walk through steps 4-9 of the checklist on the actual Windows desktop. Why:
  no CLI exists for "drag a file from Explorer to a window" or "click a menu and observe the
  visible result." Source: research.md §"Open Questions" #5.
- **D-5**: Use `target/release/notepadrs.exe` (or `target/x86_64-pc-windows-msvc/release/notepadrs.exe`
  if the cross-target path differs) — NOT debug builds. Why: SC-11 / SC-12 ship gate requires
  release artifacts; dogfood verifies the same binary that ships.
- **D-6**: Test file name MUST be `tests/dogfood_marker.rs` (NOT `dogfood_dispatch.rs` or
  similar). Why: AppCompat shim error 740 forbids `dispatch`/`install`/`update`/`setup` substrings.

### Open (executor's discretion within constraints)

- **O-1**: Whether the optional `tests/dogfood_marker.rs` ships at all. Constraint: if it does,
  it must (a) pass without GUI, (b) be deterministic, (c) not require %APPDATA% access, (d)
  use only files already in the repo (e.g. Cargo.toml itself or a controlled fixture). If it
  can't satisfy (a)-(d) cleanly, skip it — the manual checklist is the primary gate.
- **O-2**: The exact Cargo.toml edit during the manual round-trip (step 4). Suggested: append
  a single-line comment like `# notepadrs dogfood test 2026-MM-DD` near the bottom. Constraint:
  must be a syntactically-valid TOML edit (so step 7's `cargo build` doesn't fail for unrelated
  reasons).
- **O-3**: How to record the checklist results. Two options: (a) inline checkboxes in
  dogfood-checklist.md filled at execution time, or (b) a separate dogfood-result.md written
  by the executor after walking through the checklist. Recommend: (a) — keeps the artifact
  in one place.

### Considered and rejected

- **R-1**: Fully automated dogfood via Win32 SendInput / UI Automation. Rejected: high flakiness;
  re-implementing UI Automation client is v2+ work; manual marker is honest about the spirit of
  "the developer dogfoods it."
- **R-2**: Skip the dogfood marker entirely and rely on unit tests. Rejected: SC-12 explicitly
  calls for the marker; unit tests don't catch UX-level regressions (e.g. menu items wired
  to wrong handlers).
- **R-3**: Run the manual checklist BEFORE all Wave-1/2 plans complete. Rejected: D-4 — the
  marker is integration verification, must run after all features are wired (depends on 05-01,
  05-02, 05-03).

2. **tests/dogfood_marker.rs** (OPTIONAL per O-1) — If executor judges it can satisfy
   D-3 / O-1 cleanly, create:

   ```rust
   //! Plan 05-04: optional automated dogfood smoke — byte-exact round-trip on
   //! the project's own Cargo.toml without GUI interaction.
   //!
   //! This complements (does NOT replace) the manual dogfood-checklist.md. It catches
   //! the most common regression — encoding/EOL drift on save — without requiring
   //! interactive testing.

   use std::path::PathBuf;

   #[test]
   fn cargo_toml_round_trips_byte_exact_via_open_any_encoding() {
       // Phase 2 round-trip path: open_any_encoding → original_bytes cache → save_atomic
       // mirrors what notepadrs does on Ctrl+S for an unedited buffer.
       let path = project_root().join("Cargo.toml");
       let original = std::fs::read(&path).expect("read Cargo.toml");

       let opened = notepadrs::file::open_any_encoding(&path).expect("open");
       // The original_bytes cache is the verbatim file content (Phase 2 Pattern A).
       assert_eq!(opened.original_bytes, original,
           "open_any_encoding must cache original bytes byte-exact");

       // No save here — actual save requires GUI dirty flag wiring. The manual
       // checklist covers the save path. This test only proves the load path.
   }

   fn project_root() -> PathBuf {
       PathBuf::from(env!("CARGO_MANIFEST_DIR"))
   }
   ```

   If the existing `notepadrs::file::open_any_encoding` API surface or `OpenedFile` shape
   doesn't match the assertions above, ADAPT the test to whatever the actual public surface is
   — the goal is one passing test that proves Cargo.toml loads byte-exact through the
   production code path. If it can't be made to pass cleanly, SKIP this test file (per O-1).
