## Plan Decisions

(See @./.claude/pan-wizard-core/references/handoff-decisions.md for the schema.)

### Locked (executor MUST follow)

- **D-1**: Wave-0 spike runs FIRST as Task 1 and MUST commit before Task 2's production constants land. The spike writes `WRAP_ON_LPARAM` and `WRAP_OFF_LPARAM` as named constants in `src/word_wrap.rs` only after empirical confirmation. Why: research.md "Common Pitfall 1" — Microsoft's EM_SETTARGETDEVICE doc and community recipes disagree on polarity. Source: research.md §"Pattern 1" + §"Open Questions" #1.
- **D-2**: Use `EM_SETTARGETDEVICE(WPARAM=0, LPARAM=0_or_1)` PLUS a `WS_HSCROLL | ES_AUTOHSCROLL` style flip via `SetWindowLongPtrW` + `SetWindowPos(SWP_FRAMECHANGED | SWP_NOMOVE | SWP_NOSIZE | SWP_NOZORDER)`. Both are required: the message forces a re-layout; the style flip controls the scrollbar visibility. Why: research.md §"Anti-Patterns" — style-only doesn't invalidate the wrapped-line cache; message-only doesn't toggle the scrollbar. Source: research.md §"Pattern 1".
- **D-3**: New `IDM_VIEW_WORDWRAP` constant lives in Phase 5 IDM range 140-149. Specifically `IDM_VIEW_WORDWRAP = 140`. Why: Phase 1-4 used 100-135; range 140-149 is reserved for Phase 5 View additions per research.md §"File Layout". Source: research.md §"File Layout".
- **D-4**: New top-level `&View` menu placed between `&Search` and `E&ncoding` in the menu bar. Single item: `&Word Wrap` (no submenu). Why: matches Notepad++ convention; keeps Encoding menu adjacent to file-state menus. Source: research.md §"Open Questions" #3.
- **D-5**: `Tab.word_wrap` field already exists (tab.rs:42); do NOT add new field. Default `false` already set in `Tab::new_empty` at tab.rs:116. Why: pre-allocation work was done in Phase 3 to enable parallel Phase 5 development. Code Preservation Principle. Source: tab.rs grep.
- **D-6**: On tab switch (`switch_active_tab` already exists in dispatch.rs), call `apply_wrap(t.editor.hwnd_re, t.word_wrap)` for the tab being shown. Why: RichEdit may discard style state across SW_HIDE/SW_SHOW transitions. Re-applying is idempotent. Source: research.md §"Pattern 1" + Pitfall 4 (LINEFROMCHAR semantic invariance).
- **D-7**: WM_INITMENUPOPUP already handles encoding/EOL radio sync (dispatch.rs:170-207). EXTEND that block to also call `CheckMenuItem(hmenubar, IDM_VIEW_WORDWRAP as u32, if active.word_wrap { MF_CHECKED } else { MF_UNCHECKED } | MF_BYCOMMAND)`. Why: same pattern as encoding/EOL radios; only one place to update menu state. Source: research.md §"Open Questions" #3.
- **D-8**: Tests in `tests/word_wrap.rs` are PURE-LOGIC only (no HWND): test the `decide_next_wrap_state(prev: bool) -> bool` toggle helper, polarity-constant existence, default-off invariant via `Tab::new_empty` field check. No Win32 calls. Why: matches Phase 2 `dispatch_pure.rs` pattern. Test name avoids `dispatch`/`install`/`update` substrings (Pitfall: AppCompat shim error 740). Source: research.md §"Test Pitfalls".

### Open (executor's discretion within constraints)

- **O-1**: Spike test format. May use a one-shot `[[bin]]` in Cargo.toml + `[[test]]` driver, OR a manual `cargo run --example wordwrap_spike`, OR a temporary `#[ignore]`-by-default test. Constraint: must be reproducible (committed code or commit-message log lines), runnable on Windows, and produce a clear "WRAP_ON_LPARAM = X" line that the executor uses to write the production constants. Reason left open: tooling style preference; outcome is the same.
- **O-2**: Whether to add `apply_wrap` calls in `Tab::new_empty` itself or rely on Task 3's switch-active-tab re-apply. Constraint: a brand-new tab MUST end up with `word_wrap=false` AND a horizontal scrollbar visible (VIEW-07). Reason left open: either path satisfies the must-have.
- **O-3**: Menu accelerator (e.g., `Ctrl+Shift+W`). Optional. Constraint: if added, must NOT collide with existing accelerators in `build_accelerator_table` (app.rs:225+). Reason left open: SC-8 doesn't mandate an accelerator.

### Considered and rejected

- **R-1**: Toggle by destroy + recreate the RichEdit. Rejected because: would lose buffer, undo stack, scroll position, encoding metadata. (research.md §"Anti-Patterns").
- **R-2**: Style-flip without EM_SETTARGETDEVICE message. Rejected because: doesn't invalidate RichEdit's wrapped-line cache; toggle visually inconsistent across versions of msftedit.dll.
- **R-3**: Skip the Wave-0 spike and "just pick lParam=0 = wrap-on, change later if wrong." Rejected because: ships a backwards menu to the user; QA may not catch it; Pitfall 1 mitigation is mandatory per research.md.
- **R-4**: Storing word_wrap globally instead of per-tab. Rejected because: VIEW-06 explicitly says "persisted per tab"; Tab.word_wrap field already pre-allocated.

1. Extend `src/word_wrap.rs` with:

   ```rust
   use windows::Win32::Foundation::{HWND, LPARAM, WPARAM};
   use windows::Win32::UI::Controls::RichEdit::EM_SETTARGETDEVICE;
   use windows::Win32::UI::WindowsAndMessaging::{
       SendMessageW, GetWindowLongPtrW, SetWindowLongPtrW, SetWindowPos,
       GWL_STYLE, SWP_NOMOVE, SWP_NOSIZE, SWP_NOZORDER, SWP_FRAMECHANGED,
   };

   const WS_HSCROLL: u32      = 0x0010_0000;
   const ES_AUTOHSCROLL: u32  = 0x0000_0080;

   /// Pure-logic toggle helper. Returns the next state given the current one.
   /// Useful for unit testing the menu-action mapping.
   pub fn decide_next_wrap_state(prev: bool) -> bool { !prev }

   /// Apply the per-tab word-wrap setting to its RichEdit child window.
   ///
   /// Style: WS_HSCROLL + ES_AUTOHSCROLL are set when wrap is OFF (so the h-scrollbar appears
   /// and overflowing lines extend horizontally) and cleared when wrap is ON.
   /// Message: EM_SETTARGETDEVICE forces a full re-layout; polarity locked by Wave-0 spike.
   ///
   /// # Safety
   /// `hwnd_re` must be a valid RichEdit HWND owned by this thread.
   pub unsafe fn apply_wrap(hwnd_re: HWND, wrap_on: bool) {
       let mut style = GetWindowLongPtrW(hwnd_re, GWL_STYLE) as u32;
       if wrap_on {
           style &= !(WS_HSCROLL | ES_AUTOHSCROLL);
       } else {
           style |= WS_HSCROLL | ES_AUTOHSCROLL;
       }
       SetWindowLongPtrW(hwnd_re, GWL_STYLE, style as isize);
       let _ = SetWindowPos(
           hwnd_re,
           None,
           0, 0, 0, 0,
           SWP_NOMOVE | SWP_NOSIZE | SWP_NOZORDER | SWP_FRAMECHANGED,
       );
       let lparam_val = if wrap_on { WRAP_ON_LPARAM } else { WRAP_OFF_LPARAM };
       let _ = SendMessageW(
           hwnd_re,
           EM_SETTARGETDEVICE,
           Some(WPARAM(0)),
           Some(LPARAM(lparam_val)),
       );
   }
   ```

   Adjust import paths/types to match how `windows` 0.62 spells SendMessageW return type and
   the `WS_HSCROLL`/`ES_AUTOHSCROLL` symbol availability — current code in
   src/editor.rs uses raw `u32` for style bits in similar contexts.
