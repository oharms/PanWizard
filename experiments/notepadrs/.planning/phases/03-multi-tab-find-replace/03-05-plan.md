<objective>
Add the FIND-10 worker thread + epoch cancellation pattern: regex find on buffers >1MB runs on a `std::thread::spawn`-spawned worker that posts results back to the UI thread via `mpsc::Sender<FindMsg>` + `PostMessage(hwnd, WM_APP_FIND_RESULT, epoch, 0)`. Cancellation is driven by `App.find_epoch: AtomicU64` — each new query/tab-switch/dialog-close bumps the epoch; in-flight workers self-terminate on next epoch check; stale results that race past are discarded by the UI thread.

Purpose: Closes FIND-10 — the last Phase 3 requirement. Establishes the worker-thread pattern that Phase 4 will reuse for background syntax tokenization (research.md: "building it once in Phase 3 pays for itself twice"). Closes TEST-04 with worker-cancellation tests, taking the cumulative Phase 3 test count well above the orchestrator's ≥20 target.
Output: `src/find_worker.rs`, dispatch.rs's WM_APP_FIND_RESULT handler, `do_find_next`'s "if size > 1MB then spawn_find else sync path" branch, ≥6 worker-pattern tests, ≥3 integration tests.
</objective>

## Plan Decisions

(See @./.claude/pan-wizard-core/references/handoff-decisions.md for the schema.)

### Locked (executor MUST follow)

- D-1: **Worker threshold: 1 MB.** `if lf_text.len() > 1_000_000 { spawn_find(...) } else { /* sync path from 03-04 */ }`. Why: research.md Pattern 5 specifies 1MB as the threshold; matches QUAL-07 ("Regex find on a 1MB buffer (synchronous path) completes in <100ms; larger buffers use the worker path with cancellation"). Source: research.md Pattern 5 + QUAL-07.
- D-2: **Snapshot the buffer (`String::clone()`) before crossing the thread boundary.** The worker owns its `String`; UI thread is unblocked. NEVER use `Arc<Mutex<EditorState>>` or `Arc<Mutex<String>>` — both serialize UI and worker, defeating the purpose. Why: Pitfall 5 in 03-research, Anti-Pattern 8 in project research. Source: research.md Pitfall 5 + Anti-Pattern 8.
- D-3: **Epoch counter is the cancellation primitive.**
  - `App.find_epoch: AtomicU64` (already on App per Plan 03-01).
  - On every new find request: `let epoch = app.find_epoch.fetch_add(1, Ordering::Release) + 1;` — the WORKER gets epoch+1; the OLD epoch becomes stale.
  - Worker checks `epoch_atomic.load(Ordering::Acquire) == my_epoch` BEFORE each `mpsc::send` and BEFORE each `PostMessage`. If false, return early (worker self-terminates).
  - UI thread checks `wparam == app.find_epoch.load(Ordering::Acquire)` in WM_APP_FIND_RESULT; if false, drains stale messages from `find_rx` without applying.
  Why: lock-free, race-condition-proof, handles out-of-order cleanup (cancelled-then-restarted scenarios). Source: research.md Pattern 5.
- D-4: **Cancellation events that bump find_epoch:**
  - **Plan 03-02 already does:** tab switch (in WM_NOTIFY/TCN_SELCHANGE handler), close_tab_at, IDM_TAB_NEW.
  - **Plan 03-04 already does:** WM_APP_FIND_DIALOG_CLOSED.
  - **This plan adds:** new find query (do_find_next start), new replace query, edit notification (EN_CHANGE) — all bump `find_epoch.fetch_add(1, Release)` BEFORE spawning a new worker.
  - **Verification:** Plan 03-05's integration test exercises tab switch + edit + dialog close, asserting that stale results are discarded.
  Source: research.md Pitfall 9, Pattern 5.
- D-5: **Worker shares the AtomicU64 via `Arc<AtomicU64>`** — but `App.find_epoch` is currently a plain `AtomicU64`. **Action:** change `App.find_epoch: AtomicU64` to `App.find_epoch: Arc<AtomicU64>` in `src/app.rs`. Why: workers need to read the epoch from another thread; `Arc<AtomicU64>` provides shared ownership; `&AtomicU64` would require lifetime ties that can't cross `std::thread::spawn`. Source: standard Rust threading idiom.
  - **Migration cost (existing call sites — UNCHANGED):** every `app.find_epoch.fetch_add(...)` / `.load(...)` site (in dispatch.rs from Plan 03-02's tab handlers + Plan 03-04's dialog/find handlers) is unchanged because `Arc<AtomicU64>` derefs to `&AtomicU64` transparently — `Arc::deref → &AtomicU64` is a zero-cost coercion. The only site that needed reading even matters here: those calls compile and behave identically.
  - **NEW operation introduced (worker spawn site — REQUIRED):** when calling `spawn_find` for a >1MB buffer, dispatch.rs MUST execute `let epoch_atomic: Arc<AtomicU64> = Arc::clone(&app.find_epoch);` to obtain a fresh `Arc` reference to ship into the spawned thread. The clone-on-spawn is the explicit work this revision flags — without it, dispatch.rs cannot move the atomic into `std::thread::spawn`'s `move ||` closure (it would attempt to move out of `app`, which is a `&mut App`). The actual line: `let epoch_atomic = app.find_epoch.clone();` (or `Arc::clone(&app.find_epoch)` for explicitness). This is the ONLY new operation the migration introduces; everything else is structurally identical. Source: standard Rust threading idiom; clarified for revision iter 1.
- D-6: **`spawn_find` for FIND-NEXT semantics returns the FIRST hit only.** Worker calls `regex.find_iter(&snapshot)` but `break`s after the first match (or first match at-or-after the cursor for forward / first match before cursor for backward — same logic as `find::find_next`). The full match list is NOT sent over the channel. Why: FIND-10 is about Find Next on >1MB buffers, not "highlight all matches" (which is v1.x). Source: research.md Pattern 5 code example § "for find-next: stop after first hit".
- D-7: **`FindMsg::Hit { epoch, start, end, wrapped }` carries the LF-text byte offsets.** UI thread converts to UTF-16 cu via `find::utf8_to_utf16_units` BEFORE EM_EXSETSEL. **Critical:** the UI thread must use the SAME `lf_text` snapshot the worker used — but workers don't return the snapshot. **Solution:** stash the snapshot on `App.find_pending: Option<(u64, String)>` (epoch, text). When the worker posts back, UI looks up the pending snapshot, applies the hit, clears the pending. Why: avoids re-streaming/re-normalizing the buffer on the UI thread (the worker-snapshot consistency is the whole point). Source: research.md Pattern 4 + 5; new field added in this plan.
- D-8: **Worker error path:** if `regex.find_iter` panics (it shouldn't — Pike NFA is total) OR the snapshot is malformed, send `FindMsg::Error { epoch, msg }` and exit. UI thread shows status "Worker error: {msg}". Why: defensive; never hang the UI on a worker fault. Source: research.md Pattern 5 FindMsg variants (already declared in 03-01).
- D-9: **Worker thread leak on cancellation is acceptable** — when the epoch bumps mid-find, the worker eventually checks the atomic, returns early, and the OS reclaims the thread. No `JoinHandle` is kept (would force `Wait` semantics that block the UI). Why: same precedent as Phase 2's `read_with_timeout` — thread leak is bounded (one per cancelled query) and documented. Source: state.md decision "[Phase 02-file-i-o-encoding-cascade]: Worker thread leaked on timeout (bounded, acceptable for v1)".
- D-10: **`PostMessageW(hwnd, WM_APP_FIND_RESULT, WPARAM(epoch as usize), LPARAM(0))`** — the wakeup signal, **invoked by the production `wakeup` closure** passed to `spawn_find` (BLOCKER 2 / revision iter 1). WPARAM carries the epoch (u64 fits in usize on x64). LPARAM is unused (set to 0). The actual hit data flows through the mpsc channel. Why: PostMessage with payloads <`WM_USER` is unreliable across threads (research.md Pattern 5 con); the channel is the data path, PostMessage is just the wakeup. The closure injection means `find_worker.rs` does NOT call PostMessageW directly — only `dispatch.rs` does, and tests use `move |_| {}` to skip the call entirely. Source: research.md Pattern 5; closure-injection mirrors Phase 2 `read_with_timeout<F>`.
- D-11: **WM_APP_FIND_RESULT handler in dispatch.rs:**
  ```rust
  m if m == crate::app::WM_APP_FIND_RESULT => {
      let received_epoch = wparam.0 as u64;
      let current_epoch = app.find_epoch.load(Ordering::Acquire);
      if received_epoch != current_epoch {
          // Stale — drain the channel without applying.
          while let Ok(_) = app.find_rx.try_recv() {}
          return LRESULT(0);
      }
      while let Ok(msg) = app.find_rx.try_recv() {
          match msg {
              FindMsg::Hit { epoch, start, end, wrapped } if epoch == current_epoch => {
                  apply_hit_to_active_tab(app, hwnd, start, end, wrapped);
              }
              FindMsg::Done { .. } | FindMsg::Error { .. } => { /* status update */ }
              _ => {} // stale variant
          }
      }
      LRESULT(0)
  }
  ```
  `apply_hit_to_active_tab` is a helper that reads `app.find_pending` for the snapshot text, converts byte offsets to UTF-16 cu, and calls EM_EXSETSEL + EM_SCROLLCARET (mirroring the sync path in 03-04). Source: research.md Pattern 5.
- D-12: **Module exposure:** `mod find_worker;` in main.rs; `pub mod find_worker;` in lib.rs. Source: project convention.
- D-13: **`App.find_pending: Option<(u64, String)>`** new field — added in this plan to `src/app.rs`. Stores `(epoch, lf_text_snapshot)` while a worker is in flight; cleared on hit application or epoch bump. Default: `None`. Why: the UI thread needs the same byte offsets the worker computed from (D-7). Source: this plan's design.
- D-14: **Tests drive the REAL `spawn_find` via closure injection (BLOCKER 2 / revision iter 1).** `spawn_find<F>` takes `wakeup: F where F: Fn(u64) + Send + 'static`. Production callers in `dispatch.rs` pass a closure that invokes `PostMessageW(hwnd_re, WM_APP_FIND_RESULT, ...)`; test callers in `tests/find_worker.rs` pass `move |_| {}` and exercise the actual production body — epoch pre-check, `regex.find_iter`, post-work epoch re-check, `tx.send`, wakeup call. This is the same closure-injection pattern Phase 2's `read_with_timeout<F>` established. The "tab switch discards stale find" integration tests in `tests/find_replace_integration.rs` additionally mock the dispatch flow with stub state to assert the discipline at the protocol level. Why: closure injection gives REAL coverage of the production worker body without a window; full HWND-driven integration tests have no precedent in Phase 2 and add infrastructure cost disproportionate to the value. Source: research.md Plan 03-05 row; revision iter 1 BLOCKER 2.

### Open (executor's discretion within constraints)

- O-1: **Whether the worker uses `mpsc::sync_channel(N)` or `mpsc::channel()` (unbounded).** Recommendation: `channel()` (unbounded). Why: Find-Next sends at most one Hit per worker run; bounded buffering provides no value. Constraint: must be `Sender<FindMsg>` / `Receiver<FindMsg>` per Plan 03-01's App fields.
- O-2: **Whether to also use the worker for Replace All on >1MB.** Per research.md Open Question 3: NO — synchronous Replace All is fine for v1. Recommendation: leave as is (Plan 03-04's `do_replace_all` synchronous regardless of size). v1.x optimization.
- O-3: **Whether to spawn a new worker on every keystroke in the find input** (auto-find-as-you-type, Notepad++ does NOT do this — it requires explicit Find Next). Recommendation: NO — Phase 3 spec says explicit Find Next/Prev. Don't add complexity; keep `do_find_next` invoked only by IDM_FIND_NEXT/PREV.
- O-4: **Whether the integration tests run as `#[test]` in `tests/find_replace_integration.rs` or as in-source `#[cfg(test)] mod tests` in `src/find_worker.rs`.** Recommendation: external `tests/` directory — matches Plan 03-03's pattern and keeps the source files clean. Constraint: must run as `cargo test --test find_worker` and `cargo test --test find_replace_integration`.

### Considered and rejected

- R-1: **Background worker for Replace All.** Rejected: research.md Open Question 3 explicitly defers; not in 18 reqs; v1.x optimization.
- R-2: **`tokio::task::spawn_blocking`** or any async runtime. Rejected: project constraint forbids tokio.
- R-3: **`Arc<Mutex<EditorState>>` shared between UI and worker.** Rejected: Pitfall 5 — worker holds the lock while regex runs, blocking the UI on the next edit.
- R-4: **Cancellation via channel close (drop the Sender on cancel).** Rejected: a worker stuck inside `regex.find_iter` doesn't observe channel disconnect; the epoch atomic is the only mechanism that works mid-regex (research.md Pattern 5 trade-offs).
- R-5: **Polling worker status with a Win32 timer.** Rejected: PostMessage wakeup is the documented Win32 cross-thread mechanism; polling adds latency for no benefit.

```rust
//! Worker-thread + epoch-cancellation for >1MB regex find. Mirrors Phase 2's
//! `read_with_timeout` pattern (Pattern E) — std::thread::spawn + mpsc::Sender +
//! a closure-injected wakeup (production: PostMessageW; tests: no-op). Mirrors
//! Phase 2's `read_with_timeout<F>` closure-injection design (BLOCKER 2 of revision
//! iter 1) — production `dispatch.rs` callers pass a closure that invokes
//! `PostMessageW(hwnd, WM_APP_FIND_RESULT, epoch as usize, 0)`; tests pass `|_| {}`
//! so the production worker body is exercised without a window.

#![deny(clippy::unwrap_used)]
#![deny(clippy::expect_used)]

use std::sync::atomic::{AtomicU64, Ordering};
use std::sync::mpsc::Sender;
use std::sync::Arc;

// BLOCKER 2 (revision iter 1): no Win32 imports — the wakeup is closure-injected.
// Production callers in src/dispatch.rs supply a closure that captures HWND and calls
// PostMessageW(hwnd, WM_APP_FIND_RESULT, ...); test callers supply `move |_| {}`.

use crate::app::{FindMsg, FindFlags};
use crate::find;

/// Spawn a worker thread that runs `find::find_next` on the snapshot. The thread:
///   1. Compares its epoch against the shared atomic — early-exits if stale.
///   2. Runs find_next once (forward or backward, with wrap if flags.wrap).
///   3. Sends FindMsg::Hit (or Done with total=0) on the channel.
///   4. Calls the injected `wakeup(epoch)` closure to wake the UI thread (production:
///      PostMessageW; tests: no-op).
///
/// On any panic in find_next (shouldn't happen — Pike NFA is total), sends FindMsg::Error.
///
/// Worker thread leak on cancellation: when the epoch bumps mid-find, the worker
/// notices on next epoch check and returns. No JoinHandle is kept; OS reclaims the
/// thread. Acceptable for v1 (Phase 2 precedent).
pub fn spawn_find<F>(
    snapshot: String,
    regex: regex::Regex,
    cursor: usize,
    flags: FindFlags,
    epoch: u64,
    epoch_atomic: Arc<AtomicU64>,
    tx: Sender<FindMsg>,
    wakeup: F,
)
where
    F: Fn(u64) + Send + 'static,
{
    std::thread::spawn(move || {
        // Pre-check: if the epoch already moved on, abandon immediately.
        if epoch_atomic.load(Ordering::Acquire) != epoch {
            return;
        }

        // Wrap find_next in catch_unwind defensively (Pike NFA shouldn't panic, but…).
        let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
            find::find_next(&snapshot, &regex, cursor, flags)
        }));

        // Re-check epoch before sending — caller may have cancelled between the find call
        // and now (e.g. find took 50ms on a >1MB buffer; user clicked something).
        if epoch_atomic.load(Ordering::Acquire) != epoch {
            return;
        }

        let msg = match result {
            Ok(Some(hit)) => FindMsg::Hit {
                epoch,
                start: hit.start,
                end: hit.end,
                wrapped: hit.wrapped,
            },
            Ok(None) => FindMsg::Done { epoch, total: 0 },
            Err(_) => FindMsg::Error {
                epoch,
                msg: "Find worker panicked".to_owned(),
            },
        };
        if tx.send(msg).is_err() {
            // Receiver dropped (App is shutting down) — bail.
            return;
        }
        // BLOCKER 2 (revision iter 1): the side-effecting wakeup is injected as a closure.
        // Production passes `move |e| unsafe { PostMessageW(Some(hwnd), WM_APP_FIND_RESULT, WPARAM(e as usize), LPARAM(0)); }`.
        // Tests pass `move |_| {}` and exercise the real epoch-check + tx.send path above
        // without needing a window. Mirrors Phase 2's `read_with_timeout<F>` injection pattern.
        wakeup(epoch);
    });
}
```

```rust
    if let Some(hdlg) = app.find_dlg_hwnd {
        crate::find_ui::read_find_state(hdlg, &mut app.find_state);

                    crate::find::CompileError::EmptyMatch => "Pattern matches empty string".to_owned(),
                    crate::find::CompileError::Regex(re) => format!("Bad pattern: {re}"),

                crate::find_ui::set_status(hdlg, &msg);

            return;
        }

        Some(t) => t,
        None => return,

    let cr_text = match std::str::from_utf8(&cr_bytes) { Ok(s) => s, Err(_) => return };
    let lf_text = crate::eol::normalize_to_lf(cr_text);

            Some(t) => t.editor.hwnd_re,
            None => return,

            unsafe {
                use windows::Win32::Foundation::{LPARAM, WPARAM};
                use windows::Win32::UI::WindowsAndMessaging::PostMessageW;
```

```rust
/// Mirror of find_worker::spawn_find's epoch-check shape, but sends a stub message and
/// doesn't PostMessage. Used to validate cancellation discipline.
fn stub_spawn(
    epoch: u64,
    epoch_atomic: Arc<AtomicU64>,
    tx: std::sync::mpsc::Sender<StubMsg>,
    work_duration: Duration,
) -> thread::JoinHandle<()> {
    thread::spawn(move || {
        // Pre-check.
        if epoch_atomic.load(Ordering::Acquire) != epoch { return; }
        thread::sleep(work_duration);
        // Re-check after work.
        if epoch_atomic.load(Ordering::Acquire) != epoch { return; }
        let _ = tx.send(StubMsg::Hit { epoch, start: 42 });
        let _ = tx.send(StubMsg::Done { epoch });
    })
}
```

```rust
/// Simulate the cancellation primitive: an Arc<AtomicU64> shared between UI and worker.
fn fresh_epoch() -> Arc<AtomicU64> {
    Arc::new(AtomicU64::new(0))
}

#[test]
fn tab_switch_bumps_find_epoch() {
    let epoch = fresh_epoch();
    let pre = epoch.load(Ordering::Acquire);
    // Mirror dispatch.rs's switch_active_tab: fetch_add(1, Release).
    epoch.fetch_add(1, Ordering::Release);
    let post = epoch.load(Ordering::Acquire);
    assert!(post > pre, "tab switch must bump find_epoch");
}

#[test]
fn worker_with_old_epoch_self_terminates_after_tab_switch() {
    // Scenario: worker spawned at epoch 5; user switches tabs (epoch becomes 6); worker checks.
    let epoch = fresh_epoch();
    epoch.store(5, Ordering::Release);
    let worker_epoch = 5u64;
    // UI bumps epoch before worker checks.
    epoch.fetch_add(1, Ordering::Release);
    // Worker check.
    let current = epoch.load(Ordering::Acquire);
    assert_ne!(current, worker_epoch, "worker should detect stale epoch and abandon");
}

#[test]
fn ui_discards_stale_postmessage_wakeup() {
    // Scenario: stale worker posts WM_APP_FIND_RESULT(epoch=5), UI's find_epoch is 6, drain channel without applying.
    use std::sync::mpsc::channel;
    let epoch = fresh_epoch();
    epoch.store(6, Ordering::Release);
    let (tx, rx) = channel::<(u64, &'static str)>();
    let _ = tx.send((5, "stale_hit"));
    let _ = tx.send((5, "stale_done"));
    let received_wparam_epoch = 5u64;
    let current = epoch.load(Ordering::Acquire);
    let mut applied = vec![];
    if received_wparam_epoch == current {
        while let Ok(msg) = rx.try_recv() {
            applied.push(msg);
        }
    } else {
        // Stale — drain without applying.
        while let Ok(_) = rx.try_recv() {}
    }
    assert!(applied.is_empty(), "stale messages must not be applied");
}

#[test]
fn restarted_query_invalidates_old_worker() {
    // Scenario: query 1 spawns worker at epoch 1; before it returns, user types a new pattern,
    // query 2 starts (epoch 2); worker 1 must be invalidated.
    let epoch = fresh_epoch();
    let q1_epoch = epoch.fetch_add(1, Ordering::Release) + 1; // 1
    let q2_epoch = epoch.fetch_add(1, Ordering::Release) + 1; // 2
    assert_eq!(q1_epoch, 1);
    assert_eq!(q2_epoch, 2);
    // After q2 spawn, q1's epoch is now stale.
    let current = epoch.load(Ordering::Acquire);
    assert_eq!(current, 2);
    assert_ne!(current, q1_epoch);
}
```
