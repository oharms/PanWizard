<objective>

## Plan Decisions

(See @./.claude/pan-wizard-core/references/handoff-decisions.md for the schema.)

### Locked (executor MUST follow)

- D-1: **Per-tab gutter as a sibling child window.** Each Tab gets its own `gutter_hwnd: HWND` (created at the same time as `editor.hwnd_re` in `Tab::new_empty`), painted by the gutter class registered in WM_CREATE. NOT inside RichEdit's client area (RichEdit will overwrite — research.md Pattern G1 + Anti-Pattern). Why: per-tab matches Plan 03-01's per-tab RichEdit pattern; tab switch toggles visibility via `ShowWindow`, never re-queries scroll/line-count. Source: 04-research.md Pattern G1.
- D-2: **Status bar is APP-level (one), not per-tab.** `App.hstatus: HWND`. Reads from `app.tabs[app.active_tab]` on tab switch. Created in WM_CREATE via `STATUSCLASSNAME` predefined common control — auto-double-buffered (Pitfall 15 mitigation for free), auto-DPI-height. Why: research.md Pattern S1 + "Per-tab status bar" anti-pattern; one band at the bottom is the Win32 idiom. Source: 04-research.md Pattern S1.
- D-3: **Gutter window class registered exactly once at WM_CREATE.** Class atom stored as `App.gutter_class_atom: u16`. The class name is `"NotepadrsGutter"` (private — `WS_CHILD` only, no system-wide registration). All per-tab gutter HWNDs are instances of this class. Why: Win32 best practice — one RegisterClassW call per app (research.md Code Examples). Source: 04-research.md "Gutter WM_PAINT (sketch)".
- D-4: **Phase 4 syntax types live in `src/syntax/mod.rs`; per-language tokenizers (04-04a) get their own files.** This plan declares `Language`, `TokenClass`, `Token`, `LineEndState` enums + structs and `lang_from_path(&Path) -> Language` + a `tokenize_line` dispatch STUB that returns `(vec![], LineEndState::Code)` for all inputs (per-language impls land in 04-04a). Why: keeps 04-04a a pure tokenizer-implementation plan with no public-API churn; lets 04-05 (CI tests) reference the type contracts before the implementations exist. Source: 04-research.md "Pattern T1 — line-state tokenizer (per language)".
- D-5: **WM_APP_HIGHLIGHT_REQUEST = WM_USER + 12.** Declared as `pub const` in `src/app.rs`. NOT yet posted (04-04b wires it). Why: Plan 03-05 used WM_USER+10 (FIND_RESULT) and WM_USER+11 (FIND_DIALOG_CLOSED); +12 continues the sequence and keeps all WM_APP allocations in one file. Source: 04-research.md §"Recommended Phase 4 IDM / WM_APP allocations".
- D-6: **WM_SIZE four-zone layout per tab.** Replace the current `for tab in &app.tabs { MoveWindow(tab.editor.hwnd_re, 0, 0, cx, cy, true) }` with a four-zone layout:
  - Tab strip: `MoveWindow(app.htabs, 0, 0, cx, TAB_STRIP_HEIGHT, true)` — already done by 03-02; this plan does NOT touch the tab-strip layout call (assumes it's there).
  - Status bar: status bar self-positions on its own `WM_SIZE` from the parent's resize — but we can also explicitly call `SendMessageW(app.hstatus, WM_SIZE, 0, 0)` to force re-layout. After parent WM_SIZE completes, `GetWindowRect(app.hstatus)` gives the actual rect; cache `app.status_height` once at WM_CREATE, refresh on DPI change.
  - Gutter (per tab): `MoveWindow(tab.gutter_hwnd, 0, TAB_STRIP_HEIGHT, tab.gutter_width, cy - TAB_STRIP_HEIGHT - app.status_height, true)`.
  - RichEdit (per tab): `MoveWindow(tab.editor.hwnd_re, tab.gutter_width, TAB_STRIP_HEIGHT, cx - tab.gutter_width, cy - TAB_STRIP_HEIGHT - app.status_height, true)`.
  Use a default `tab.gutter_width = 40` (4-digit lines × 8px em + 8px padding) for now; 04-02 wires the dynamic width algorithm. Why: research.md Pattern G1 + S1 + the locked-decision "Layout discipline" in 04-research.md user_constraints. Source: 04-research.md Pattern G1 layout, P4-2.
- D-7: **`Tab::new_empty` extension** — accept the gutter class atom as a third parameter (e.g., `Tab::new_empty(parent, id, initially_visible, gutter_class_atom)`); create the gutter HWND inside `new_empty` via `CreateWindowExW(WS_CHILD | WS_VISIBLE, gutter_class_atom_to_str, ..., parent, ...)` immediately after `EditorState::create`. **Important:** the gutter HWND's `GWLP_USERDATA` is set to the sibling RichEdit's `hwnd_re` so the gutter's own WM_PAINT can call `EM_GETFIRSTVISIBLELINE` on it (04-02). Hide the gutter via `ShowWindow(SW_HIDE)` if `initially_visible == false` (mirrors RichEdit visibility). Tab's Drop impl extended to `DestroyWindow(self.gutter_hwnd)` BEFORE `DestroyWindow(self.editor.hwnd_re)`. Why: keeps the per-tab construction atomic; ensures GDI handle leak symmetry with the existing RichEdit Drop. Source: 04-research.md Pattern G1, Pitfall 2 (Phase 3) precedent.
- D-8: **`status_height` and `line_height_px` are cached at WM_CREATE.** After `create_status_bar` returns, call `GetWindowRect(hstatus)` and store `bottom - top` as `app.status_height`. After the first tab is pushed (so an HFONT exists), open a memory DC, `SelectObject(font)`, `GetTextMetricsW`, store `tm.tmHeight + tm.tmExternalLeading` as `app.line_height_px`. These are read by 04-02 (gutter) and used by 04-04b (highlight) — pre-computing here means later plans don't have to. Why: avoids per-paint metric queries; line height is invariant within a session (font is fixed). Source: 04-research.md "Don't Hand-Roll" — line height cache.
- D-9: **lang_from_path is the simplest possible match.** No regex, no trie, no caching:
  ```rust
  pub fn lang_from_path(path: &Path) -> Language {
      let ext = path.extension().and_then(OsStr::to_str).map(str::to_ascii_lowercase);
      match ext.as_deref() {
          Some("js") | Some("mjs") => Language::JavaScript,
          Some("py")               => Language::Python,
          Some("json")             => Language::Json,
          Some("md") | Some("markdown") => Language::Markdown,
          _                        => Language::Plain,
      }
  }
  ```
  Why: 6 extensions match the SYNTAX requirements verbatim; trivially testable without Win32; ~10 LOC. Source: 04-research.md "Don't Hand-Roll" — extension match.
- D-10: **Pure-logic tests in `tests/phase04_data_shape.rs`** cover the `lang_from_path` matrix (every extension + UPPERCASE + no extension + multi-component path). NO Win32 imports — must be a `cargo test --lib` clean module. Why: matches Phase 2's `dispatch_pure.rs` precedent and Phase 3's `tab_model.rs` precedent — pure-logic tests stay headless. Source: 04-research.md Pattern V1.
- D-11: **`set tab.language` lives in `do_file_open`.** After `EditorState::open_text_with_metadata` succeeds AND after the file path is committed to `tab.editor.current_path`, call `tab.language = lang_from_path(&path)`. NEW EMPTY tabs get `Language::Plain` (default). The actual highlight render is wired in 04-04b; this plan only sets the field. Why: keeps language detection at the file-open boundary; new tabs without a path can't have a language. Source: 04-research.md user_constraints "Where extension→language detection lives".
- D-12: **Lock the `lib.rs` / `main.rs` mod-declaration ordering.** This addresses the Warning #5 hand-wave. Concretely:
  - **Task 1** creates `src/syntax/mod.rs` AND the five sibling stub files (`src/syntax/{plain,json,markdown,javascript,python}.rs`) AND adds **only** `pub mod syntax;` to `src/lib.rs` (and `mod syntax;` to `src/main.rs`). Task 1 does NOT touch `gutter` or `status_bar` declarations.
  - **Task 2** creates `src/gutter.rs` and `src/status_bar.rs`, AND in the same task adds `pub mod gutter; pub mod status_bar;` to `src/lib.rs` (and `mod gutter; mod status_bar;` to `src/main.rs`).
  - **No placeholder one-line files.** The build is green at the end of each task because each module declaration lands together with the file it declares. Why: removes the cross-task "lib.rs declares a module whose file doesn't exist yet" risk that the original Task 1 done-block hand-waved with "alternatively, hold the lib.rs changes until Task 2." Source: plan-checker Warning #5.
- D-13: **EM_SETEVENTMASK pre-allocation.** Set the FULL Phase-4 event mask in `src/editor.rs::EditorState::create` IN THIS PLAN: `EM_SETEVENTMASK = ENM_CHANGE | ENM_SCROLL | ENM_SELCHANGE`. Phase 1's current line is `LPARAM(ENM_CHANGE as isize)`; Plan 04-01 Task 3 changes it to `LPARAM((ENM_CHANGE | ENM_SCROLL | ENM_SELCHANGE) as isize)`. **Neither Plan 04-02 nor Plan 04-03 mutates this line.** They only register WM_NOTIFY handlers (EN_VSCROLL in 04-02; EN_SELCHANGE in 04-03). Why: eliminates the wave-2 file-collision on `src/editor.rs` between 04-02 and 04-03 (plan-checker Blocker #2). Both downstream plans can run truly parallel in Wave 2 because neither touches editor.rs. Source: plan-checker Blocker #2 fix; 04-research.md Pitfall P4-2 (ENM_SCROLL needed for Page Down) + Pattern S1 (ENM_SELCHANGE needed for cursor moves).

### Open (executor's discretion within constraints)

- O-1: **Gutter class atom storage.** Either `App.gutter_class_atom: u16` (returned by `RegisterClassW`) or a `static AtomicU16` set on first call. Recommendation: App field is cleaner — survives unloads, matches existing per-app handles like `haccel`. Constraint: must be set BEFORE `push_empty_tab` is called in WM_CREATE.
- O-2: **Where `register_gutter_class` lives.** Either in `src/gutter.rs` (preferred — keeps gutter ownership local) or in `src/main.rs` next to the main window class. Recommendation: `src/gutter.rs` — module is small enough.
- O-3: **Initial `tab.gutter_width` default value.** Either hardcoded 40px or `4 * em_width + 8` computed once at WM_CREATE. Recommendation: hardcoded 40px (well within the typical em-width × 4-digits + 8 padding range). 04-02 makes this dynamic.
- O-4: **Whether to add `pub mod plain; pub mod json; pub mod markdown; pub mod javascript; pub mod python;` to `src/syntax/mod.rs` now (as empty stubs) or wait for 04-04a.** Recommendation: declare all five as `pub mod` lines in `src/syntax/mod.rs` and create files with a STUB tokenize_line per module — guarantees 04-04a doesn't have to touch `mod.rs`. The `tokenize_line` dispatch in `mod.rs` calls each submodule.

### Considered and rejected

- R-1: **Single shared gutter that re-renders on tab switch.** Rejected: forces gutter to re-query line count + scroll position on every switch (P4-2 P4-4 risk surface area grows); per-tab matches Plan 03-01 RichEdit per-tab discipline. Source: 04-research.md Pattern G1 "Why per-tab".
- R-2: **Putting per-language tokenizers under `src/syntax_*.rs` (flat, no subdir).** Rejected: 5 tokenizers + dispatch + helpers would clutter `src/`; `src/syntax/` is the project's already-suggested layout (idea.md "src/syntax/{js,py,json,md}.rs"). Source: idea.md Constraints.
- R-3: **Hardcoding status bar height to 22px.** Rejected: DPI-dependent; STATUSCLASSNAME self-determines and exposes its own rect. Source: 04-research.md Anti-Pattern "Hardcoding status-bar height".
- R-4: **Computing `lang_from_path` lazily in `tokenize_line`.** Rejected: each `tokenize_line` call would re-parse the path; storing on `Tab` means file open does it once, edits read the cached value.
- R-5: **Setting EM_SETEVENTMASK incrementally across plans (each plan ORs in its own bit).** Rejected: causes wave-2 collision on editor.rs between 04-02 and 04-03 (plan-checker Blocker #2). Pre-allocating the full mask here makes the downstream plans true parallel candidates. Source: plan-checker Blocker #2.

//! Phase 4 syntax-highlighting type system + per-language dispatch.
//!
//! Plan 04-01 declares:
//!   - `Language` enum (5 variants)
//!   - `TokenClass` enum (11 variants — superset across all languages)
//!   - `Token` struct (start/end byte offsets within a line + class)
//!   - `LineEndState` enum (carries multi-line state across line boundaries)
//!   - `lang_from_path(&Path) -> Language` — extension dispatch
//!   - `tokenize_line(...)` STUB returning `(Vec::new(), LineEndState::Code)` for all inputs
//!
//! Plan 04-04a implements per-language `tokenize_line` impls in the submodules.
//! Plan 04-05 ships golden-token-stream tests in `tests/syntax_tokenizers.rs`.

#![deny(clippy::unwrap_used)]
#![deny(clippy::expect_used)]

use std::ffi::OsStr;
use std::path::Path;

pub mod plain;
pub mod json;
pub mod markdown;
pub mod javascript;
pub mod python;

#[derive(Clone, Copy, Debug, PartialEq, Eq, Default)]
pub enum Language {
    #[default]
    Plain,
    Json,
    Markdown,
    JavaScript,
    Python,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum TokenClass {
    Plain, Keyword, String, Number, Comment, Punct,
    Heading, Bold, Italic, CodeFence, Link,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub struct Token {
    pub start: u32, // byte offset within the line
    pub end: u32,   // byte offset within the line, exclusive
    pub class: TokenClass,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum PyTripleKind {
    SingleQuote, // '''
    DoubleQuote, // """
}

#[derive(Clone, Copy, Debug, PartialEq, Eq, Default)]
pub enum LineEndState {
    #[default]
    Code,
    JsBlockComment,
    JsTemplateLiteral { backticks: u8 }, // tagged-template depth (v1: 0 or 1)
    PyTripleString { kind: PyTripleKind },

/// Detect the language for a file from its extension.
/// Returns `Language::Plain` for unknown / missing extensions.
/// Case-insensitive on the extension component.
pub fn lang_from_path(path: &Path) -> Language {
    let ext = path
        .extension()
        .and_then(OsStr::to_str)
        .map(str::to_ascii_lowercase);
    match ext.as_deref() {
        Some("js") | Some("mjs") => Language::JavaScript,
        Some("py") => Language::Python,
        Some("json") => Language::Json,
        Some("md") | Some("markdown") => Language::Markdown,
        _ => Language::Plain,
    }
}

/// Create the status bar (one per main window). Auto-positions at the bottom and
/// auto-determines its height per current DPI. Caller stores the returned HWND

pub unsafe fn create_status_bar(parent: HWND) -> Result<HWND> {

        WINDOW_EX_STYLE(0),

        w!(""),
        WS_CHILD | WS_VISIBLE | WINDOW_STYLE(SBARS_SIZEGRIP as u32),
        0, 0, 0, 0,
        Some(parent),
        None,
        Some(hinstance),
        None,
    )?;

        hstatus,
        SB_SETPARTS,
        Some(WPARAM(SEGMENT_PARTS.len())),
        Some(LPARAM(SEGMENT_PARTS.as_ptr() as isize)),

        SB_SETTEXTW,
        Some(WPARAM(idx as usize)),

}
