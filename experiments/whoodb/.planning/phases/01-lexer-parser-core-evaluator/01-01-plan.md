<grammar>
<!-- Locked grammar surface for Phase 1. -->
<!-- Precedence (LOW to HIGH): OR < AND < NOT < comparison < BETWEEN/IN/LIKE < primary -->

```
SelectStmt   := SELECT ProjectionList FROM FromTarget [WHERE OrExpr] EOF

ProjectionList := '*'
              | ProjectionItem (',' ProjectionItem)*

ProjectionItem := ColumnRef ['AS' IDENT]

ColumnRef    := IDENT ('.' IDENT)*    // dot-path

FromTarget   := STRING | IDENT         // single file path (Phase 1) — string literal OR bareword

OrExpr       := AndExpr ('OR' AndExpr)*
AndExpr      := NotExpr ('AND' NotExpr)*
NotExpr      := 'NOT' NotExpr | Comparison
Comparison   := Primary [ComparisonRest]
ComparisonRest := ('=' | '!=' | '<' | '<=' | '>' | '>=') Primary
              | 'BETWEEN' Primary 'AND' Primary       // 'AND' is consumed HERE, not at AndExpr
              | 'IN' '(' Literal (',' Literal)* ')'
              | 'LIKE' STRING
Primary      := Literal | ColumnRef | '(' OrExpr ')'
Literal      := NUMBER | STRING | 'true' | 'false' | 'null'
```

Rules embedded in the grammar:
- Keyword recognition is case-insensitive; lexer normalizes KEYWORD `value` to upper-case.
- BETWEEN MUST be parsed at `Comparison` level so its inline `AND` is consumed before `AndExpr` runs.
- `NOT` is right-associative unary; it sits above `AND`/`OR` and below `Comparison`.
- `IN` requires literals only in Phase 1 (no sub-queries, no field lists — keeps grammar small).
- `LIKE` accepts a STRING literal pattern only; the regex compilation lives in `where.js` (Plan 01-02), not the parser.
- `*` is a separate WILDCARD token type, not an OPERATOR or IDENT — the lexer must distinguish it.
</grammar>

<pitfall_mitigations>
<!-- These are NOT suggestions. Each task action explicitly references the relevant pitfall. -->

1. **Left-recursion (Pitfall 1):** Every expression-level parser function uses an iterative `while` loop, NOT a recursive call after consuming the operator. Pattern:
   ```js
   function parseAndExpr(state) {
     let left = parseNotExpr(state);
     while (peek(state)?.type === 'KEYWORD' && peek(state).value === 'AND') {
       consume(state);
       const right = parseNotExpr(state);
       left = { type: 'BinaryOp', op: 'AND', left, right };
     }
     return left;
   }
   ```
   Verify with a smoke timer in `test/parser.test.js` that asserts `parse(tokenize('SELECT * FROM f WHERE a = 1 AND b = 2 AND c = 3 AND d = 4'))` returns in <50ms (generous; should be sub-millisecond).

2. **BETWEEN/AND collision (Pitfall 2):** BETWEEN is parsed at the `Comparison` level. After matching `BETWEEN`, parse the lower bound, then `expect('AND')` (consuming it as a SEPARATOR, not as boolean AND), then parse the upper bound. The `AndExpr`-level `while (peek === 'AND')` will only see the AND that follows the BETWEEN's upper bound and beyond — it won't steal the inner one. Test exactly: `WHERE n BETWEEN 1 AND 10 AND name = 'x'`.

3. **NOT precedence (Pitfall 7):** Place `parseNotExpr` ABOVE `parseAndExpr` and BELOW `parseComparison` in the precedence ladder. Document the ladder as a comment block at the top of `parser.js` BEFORE writing any expression rule.

4. **Lexer column tracking (Pitfall 8):** `col` is 1-indexed and equals the (1-indexed) position of the token's FIRST character. Compute it at the START of each token-extraction iteration, BEFORE consuming any chars. Off-by-one is the top bug here — the test `tokenize('SELECT FROM')` MUST produce a token with `value === 'FROM' && col === 8`.

5. **Keyword case-insensitivity (Pitfall in standards):** `tokenize('select')`, `tokenize('SELECT')`, `tokenize('SeLeCt')` all produce a Token with `type === 'KEYWORD'` and `value === 'SELECT'`. Internal value is normalized to upper-case by the lexer so the parser can switch on it without re-uppercasing.

6. **PARSE-09 GROUP BY rule (semantic, not syntactic):** Lives in `analyze.js`, not the parser. Walks the AST looking for a `GROUP BY` clause; if present, every projected ColumnRef must either appear in the GROUP BY column list OR be wrapped in an Aggregate node (Aggregate is a Phase-2 type, but the analyzer's logic is written now and just sees zero Aggregates in Phase 1 — so any GROUP BY query in Phase 1 with non-grouped non-aggregate columns errors at parse time). The Phase 1 test exercises this with a hand-built AST (since the parser doesn't yet emit GROUP BY nodes — it just rejects them) OR with a parser that DOES emit GROUP BY nodes. **Planner choice:** the parser DOES emit GROUP BY nodes (cheap to add) and the analyzer enforces the rule. This means PARSE-09 is testable end-to-end with `parse() -> analyze()` in Plan 01-01. The evaluator in Plan 01-02 will refuse to execute a plan with `groupBy !== null` (returns a "GROUP BY not supported until Phase 2" error). That deferral is documented in 01-02's task action; Phase 1 only needs the analyzer rule to fire.
</pitfall_mitigations>
</context>

  <name>Task 2: Implement lexer + lexer tests (PARSE-01..04, partial PARSE-08)</name>
  <files>src/lexer.js, test/lexer.test.js</files>
  <action>
    Build `tokenize(query: string) => Token[]` plus a comprehensive test file. The lexer is the project's single highest-leverage test target — get every token type, every literal form, every position case right here, and parser failures downstream become isolated to grammar logic.

    **Algorithm:**
    Walk the query string with an index `i`. At the start of each iteration, capture `col = i + 1` (1-indexed). Skip whitespace (advance `i`, do not emit a token, continue — but still update `col` on the next iteration). For each non-whitespace start, dispatch by the first character:

    1. **String literal:** starts with `'`. Consume up to the next unescaped `'`. Phase 1 supports the simplest form — no escape sequences, no double-`'`-as-escape. If the closing quote is missing before EOF, throw `LexError('unterminated string', col)`. Token: `{ type: 'STRING', value: <inner text>, col }`.
    2. **Number literal:** starts with a digit (or `-` followed by a digit — but Phase 1 keeps it simple: handle `-` as a unary operator at the parser level, NOT here. So the lexer only emits unsigned numbers). Match `/^\d+(\.\d+)?/`. Token: `{ type: 'NUMBER', value: parseFloat(matched), col }`.
    3. **Identifier or keyword:** starts with a letter or underscore. Match `/^[A-Za-z_][A-Za-z0-9_.]*/` — note the inclusion of `.` so dotted identifiers like `usage.input_tokens` are a single IDENT token (per `01-research.md` recommendation 1). After matching, check if the upper-cased form is a reserved keyword: `SELECT, FROM, WHERE, AND, OR, NOT, IN, LIKE, BETWEEN, AS, GROUP, BY, ORDER, LIMIT, OFFSET, ASC, DESC, TRUE, FALSE, NULL`. If yes:
       - For `TRUE`/`FALSE`: emit `{ type: 'BOOLEAN', value: <true|false>, col }`.
       - For `NULL`: emit `{ type: 'NULL_LIT', value: null, col }`.
       - Otherwise: emit `{ type: 'KEYWORD', value: <UPPERCASED>, col }`.
       If not a keyword: emit `{ type: 'IDENT', value: <ORIGINAL — preserves case>, col }`.
       **Critical:** the IDENT value is the original case (because field names in JSONL rows ARE case-sensitive — `usage.input_tokens` and `Usage.Input_Tokens` are different keys).
    4. **Wildcard:** `*` as a standalone token (not part of an identifier). Token: `{ type: 'WILDCARD', value: '*', col }`.
    5. **Operators:** match longest-first: `<=`, `>=`, `!=`, then `=`, `<`, `>`. Token: `{ type: 'OPERATOR', value: <matched>, col }`.
    6. **Punctuation:** `(`, `)`, `,`. Token: `{ type: 'PUNCT', value: <char>, col }`.
    7. **Anything else:** throw `LexError(\`unexpected character '\${char}'\`, col)`.

    After the main loop, push `{ type: 'EOF', value: null, col: query.length + 1 }`.

    **Pitfall encoded in code (do not skip):** at the START of each iteration, before any consumption, set `const col = i + 1`. NEVER compute col after advancing i. This is Pitfall 8 from research/pitfalls.md and is the top off-by-one source.

    **Reserved keyword list:** define as a `Set` at the top of the file. Use `.has(value.toUpperCase())` for the lookup. Do NOT inline the list at every check.

    **Exports:** `export function tokenize(query)`. That's the only export.

    ### A. src/parser.js

    **Structure:** A `parse(tokens)` entry point plus mutually recursive helpers, each receiving and returning a `state` object `{ tokens, pos }` (pass-by-reference is fine; state.pos is mutated). Provide internal helpers:
    - `peek(state, offset = 0)` → `state.tokens[state.pos + offset]` or undefined
    - `consume(state)` → returns the current token and advances pos by 1
    - `expect(state, type, value?)` → if peek doesn't match, throw `ParseError`; else consume and return the token. The error message format is exactly `expected <X>, got '<Y>'` where X is the expected token description and Y is the actual token's `value` (or `<EOF>` if EOF). The col is the actual token's col.

    **Top-of-file comment block — write THIS BEFORE any rule code (Pitfall 7 mitigation):**
    ```
    // Operator precedence ladder (LOW to HIGH):
    //   OR
    //   AND
    //   NOT      <- unary, above AND/OR
    //   comparison (=, !=, <, <=, >, >=, BETWEEN, IN, LIKE)
    //   primary (literal, column, parenthesised expr)
    //
    // Implementation note: each precedence level is an iterative `while` loop,
    // NOT a recursive call after consuming an operator. This avoids left
    // recursion (the project's #1 documented pitfall — see research/pitfalls.md).
    ```

    **Grammar functions (one per rule):**
    - `parseSelect(state)` → consumes KEYWORD SELECT, parses ProjectionList, consumes KEYWORD FROM, parses FromTarget, optionally parses WHERE clause, then GROUP BY clause, then EOF check. Returns `selectStmt(...)`. NOTE: ORDER BY / LIMIT / OFFSET are NOT parsed in Phase 1 — if encountered, throw `ParseError("ORDER BY/LIMIT/OFFSET not supported in Phase 1", col)`. (Phase 2 will fill these in.)
    - `parseProjectionList(state)` → handles `*` (returns `[wildcard()]`) OR a comma-separated list of `parseProjectionItem` results.
    - `parseProjectionItem(state)` → parses a ColumnRef (a single IDENT), then optionally consumes `KEYWORD AS` followed by an IDENT, wrapping in `alias(name, columnRef)`. Returns the bare ColumnRef if no AS clause.
    - `parseFromTarget(state)` → consumes either an IDENT or a STRING token, returns the string value. (Bare identifiers are allowed because users may write `FROM logs` without quoting; STRING is allowed for paths with spaces or extensions like `'data.jsonl'`.)
    - `parseGroupBy(state)` → if peek is KEYWORD GROUP, consume it, expect KEYWORD BY, parse a comma-separated list of ColumnRefs. Returns the array. Else returns `null`. (Wired in Phase 1 SOLELY so the analyzer can run PARSE-09; the evaluator in Plan 01-02 will refuse to execute groupBy !== null.)
    - `parseWhere(state)` → if peek is KEYWORD WHERE, consume it, return `parseOrExpr(state)`. Else null.
    - `parseOrExpr(state)`, `parseAndExpr(state)`, `parseNotExpr(state)`, `parseComparison(state)`, `parsePrimary(state)` — the precedence ladder. Each uses an iterative `while` (or for NOT, a single recursive descent for right-associativity).

    **parseComparison MUST handle BETWEEN at this level (Pitfall 2 mitigation):**
    ```js
    function parseComparison(state) {
      const left = parsePrimary(state);
      const next = peek(state);
      if (!next) return left;
      // BETWEEN — consume the inline AND HERE so it never reaches parseAndExpr
      if (next.type === 'KEYWORD' && next.value === 'BETWEEN') {
        consume(state);
        const lo = parsePrimary(state);
        expect(state, 'KEYWORD', 'AND');   // SEPARATOR — consumed at this level
        const hi = parsePrimary(state);
        return betweenRange(left, lo, hi);
      }
      // IN
      if (next.type === 'KEYWORD' && next.value === 'IN') {
        consume(state);
        expect(state, 'PUNCT', '(');
        const values = [parseLiteral(state)];
        while (peek(state)?.type === 'PUNCT' && peek(state).value === ',') {
          consume(state);
          values.push(parseLiteral(state));
        }
        expect(state, 'PUNCT', ')');
        return inList(left, values);
      }
      // LIKE
      if (next.type === 'KEYWORD' && next.value === 'LIKE') {
        consume(state);
        const pat = expect(state, 'STRING');
        return likePattern(left, pat.value);
      }
      // Comparison operators
      if (next.type === 'OPERATOR') {
        consume(state);
        const right = parsePrimary(state);
        return binaryOp(next.value, left, right);
      }
      return left;
    }
    ```
    Where `parseLiteral(state)` consumes one of NUMBER, STRING, BOOLEAN, NULL_LIT and returns a `literal()` node.

    **parsePrimary handles parens, literals, column refs:**
    - If peek is `PUNCT '('`: consume, parse OrExpr, expect `PUNCT ')'`, return the inner expr.
    - If peek is one of NUMBER/STRING/BOOLEAN/NULL_LIT: return `literal(token.value)`.
    - If peek is IDENT: consume and return `columnRef(token.value)`.
    - Else: throw ParseError with the actual token's col.

    **Exports:** `export function parse(tokens)`. That's the only export. `parse` is a thin wrapper that creates `state = { tokens, pos: 0 }`, calls `parseSelect(state)`, and confirms peek is EOF.

    ### B. src/analyze.js

    Single function `analyze(ast: SelectStmt) => ExecutionPlan`.

    **Behavior:**
    - Returns `{ scan: { source: ast.from }, filter: ast.where, projection: ast.projection, groupBy: <ast.groupBy or null>, sort: null, limit: null }`.
    - **PARSE-09 rule:** if `ast.groupBy !== null`, walk every ProjectionItem and check: every ColumnRef in the projection must EITHER appear in `ast.groupBy` (matched by `name` string equality) OR be wrapped in an Aggregate node. Aggregate is a Phase-2 type, so in Phase 1 we treat "is this an Aggregate?" as "does the node's type === 'Aggregate'?" — which will always be false in Phase 1, so the rule reduces to: every projected ColumnRef must appear in groupBy. If any doesn't, throw `ParseError('column "X" must appear in GROUP BY or be wrapped in an aggregate', 0)` (col 0 because we lost source position by this point — the analyzer is post-parse).
    - Wildcard projection (`*`) with GROUP BY is also an error: throw `ParseError('SELECT * is not allowed with GROUP BY', 0)`.
    - Aliases: if a ProjectionItem is an Alias wrapping a ColumnRef, check the inner ColumnRef.

    **Exports:** `export function analyze(ast)`.

    **PROJ-04 negative half (alias not in WHERE) — defer to where.js (Plan 01-02).** The analyzer cannot easily check this without walking the WHERE expression for ColumnRefs and matching against alias names. Plan 01-02 will add the check inside `where.js` OR keep the rule "WHERE only sees raw row fields, never aliases" implicit in the architecture (since `where.js` evaluates against the raw row before projection runs). Document this trade in 01-02.
