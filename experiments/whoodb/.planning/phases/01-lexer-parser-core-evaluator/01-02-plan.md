<null_semantics_policy>
**LOCKED — write THIS exactly as a comment block at the top of src/where.js BEFORE any comparison code:**

```
// =============================================================================
// NULL semantics — two-valued logic policy
// =============================================================================
//
// Standard SQL uses three-valued logic where NULL = NULL evaluates to NULL
// (unknown), not true. This project deviates: any comparison involving NULL
// (or undefined, treated identically) returns false.
//
// Rationale: JSONL logs are inherently nullable; propagating NULL through
// expression trees adds complexity without adding query power for the log-
// search use case. Two-valued logic is documented and predictable.
//
// Concrete consequences:
//   WHERE missing_field = null         -> 0 rows
//   WHERE missing_field = 'anything'   -> 0 rows
//   WHERE n > null                     -> 0 rows
//   WHERE NOT (missing_field = 'x')    -> 0 rows  (because (missing = 'x') is false, NOT false is true,
//                                                   BUT we still don't have a way to filter on missing-ness;
//                                                   that requires `IS NOT NULL`, which is V2.)
//
// Implementation: every comparison helper (sqlEqual, sqlLessThan, sqlBetween,
// sqlIn, sqlLike) MUST short-circuit to false when ANY operand is null OR
// undefined. NEVER use === for SQL equality.
// =============================================================================
```

This block is not optional. It belongs in the source file because anyone modifying the WHERE evaluator later needs to see it before they touch comparison logic.
</null_semantics_policy>

<like_anchoring_policy>
**LOCKED — implement EXACTLY this LIKE-to-regex conversion (Pitfall 5 mitigation):**

```js
function likeToRegex(pattern) {
  // Escape every regex metacharacter EXCEPT %
  const escaped = pattern.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
  // Replace SQL % with .* (after escaping, so the % is still a literal % at this point)
  const regexBody = escaped.replace(/%/g, '.*');
  // Anchor at BOTH ends — this is what makes 'err%' NOT match 'my_error'
  return new RegExp('^' + regexBody + '$');
  // Note: case-sensitive (no `i` flag). JSONL field values are arbitrary strings;
  // case-sensitive LIKE avoids surprising log-query results.
}
```

Test gate (Pitfall 5 from research/pitfalls.md, also Phase 1 Success Criterion #4):
- `'err%'` against `'my_error'` returns false (anchored at both ends, my_ prefix is not a match).
- `'err%'` against `'error'` returns true.
- `'%error'` against `'my_error'` returns true.
- `'%error%'` against `'my_error_log'` returns true.
</like_anchoring_policy>

<pitfall_mitigations>
1. **NULL semantics (Pitfall 4):** Two-valued logic, documented at top of where.js. Implement `sqlEqual`, `sqlCompare`, `sqlBetween`, `sqlIn`, `sqlLike` helpers that all return false when any operand is null/undefined.
2. **LIKE anchoring (Pitfall 5):** Locked to the exact `likeToRegex` implementation above.
3. **Stream backpressure (Pitfall 6):** Use `for await (const line of rl)` — NOT `rl.on('line')`. Locked by 01-context.md and research/summary.md.

    **Internal helpers (NOT exported, used internally):**

    1. `resolvePath(row, path)` — walks `row[path[0]]?.[path[1]]?.[path[2]]...` and returns the leaf value, or `undefined` if any step is null/undefined. THIS is the dot-nested access (PROJ-02 reused for WHERE evaluation). Implementation:
       ```js
       function resolvePath(row, path) {
         let cur = row;
         for (const key of path) {
           if (cur == null) return undefined;
           cur = cur[key];
         }
         return cur;
       }
       ```
       Note: `cur == null` covers BOTH null AND undefined (intentional — see NULL policy).

    2. `sqlEqual(a, b)` — returns false if either is null/undefined; else `a === b`.
    3. `sqlCompare(op, a, b)` — for `<`, `<=`, `>`, `>=`. Returns false if either is null/undefined OR not a number. Else applies the JS operator.

    6. `sqlLike(v, pattern)` — returns false if v is null/undefined or not a string. Else `likeToRegex(pattern).test(v)`. The likeToRegex helper is LOCKED in `<like_anchoring_policy>` above.
    7. `likeToRegex(pattern)` — exact implementation locked in `<like_anchoring_policy>`.

    **Main exported function: `evalWhere(node, row)`**

    Walks the AST node and returns a boolean. Switch on `node.type`:

    ```js
    export function evalWhere(node, row) {
      if (node === null || node === undefined) return true;   // null WHERE means no filter

      switch (node.type) {
        case 'BinaryOp': {
          const { op, left, right } = node;
          if (op === 'AND') return evalWhere(left, row) && evalWhere(right, row);
          if (op === 'OR')  return evalWhere(left, row) || evalWhere(right, row);
          // Comparison operators — evaluate both sides to values
          const lv = evalValue(left, row);
          const rv = evalValue(right, row);
          if (op === '=')  return sqlEqual(lv, rv);
          if (op === '!=') return !sqlEqual(lv, rv);   // !=: false if either is null (NULL policy still applies — see test)
          if (op === '<' || op === '<=' || op === '>' || op === '>=') return sqlCompare(op, lv, rv);
          throw new Error(`unknown BinaryOp: ${op}`);
        }
        case 'UnaryOp': {
          if (node.op === 'NOT') return !evalWhere(node.expr, row);
          throw new Error(`unknown UnaryOp: ${node.op}`);
        }
        case 'BetweenRange': {
          return sqlBetween(evalValue(node.expr, row), evalValue(node.lo, row), evalValue(node.hi, row));
        }
        case 'InList': {
          return sqlIn(evalValue(node.expr, row), node.values);
        }
        case 'LikePattern': {
          return sqlLike(evalValue(node.expr, row), node.pattern);
        }
        default:
          throw new Error(`evalWhere: unsupported node type ${node.type}`);
      }
    }
    ```

    **`evalValue(node, row)`** — internal helper that resolves a value-producing node (Literal or ColumnRef) to its JS value. ColumnRef uses `resolvePath(row, node.path)`. Literal returns `node.value`. Throw on any other type.

    **Important on `!=` and NULL:** under two-valued logic, `null != 5` is FALSE (because `sqlEqual(null, 5)` is false; `!sqlEqual` would be true — BUT this would say "null is not equal to 5" returns true, which propagates NULL into a positive match). For consistency with the policy ("any comparison involving NULL returns false"), `!=` is special-cased: if either side is null/undefined, return FALSE outright (not !sqlEqual). Test this explicitly.

    **Exports:** `export function evalWhere(node, row)`. Internal helpers (`resolvePath`, `sqlEqual`, etc.) are NOT exported (project.js will export its own copy of resolvePath — keeps the modules independent for unit testing).

    ```js
    export function project(projectionList, row) {
      // Wildcard: shallow copy
      if (projectionList.length === 1 && projectionList[0].type === 'Wildcard') {
        return { ...row };
      }

      const out = {};
      for (const item of projectionList) {
        if (item.type === 'ColumnRef') {
          // Output key is the ORIGINAL dotted name (e.g. 'usage.output_tokens'),
          // matching SQL's behavior where SELECT a.b returns a column named "a.b".
          out[item.name] = resolvePath(row, item.path);
        } else if (item.type === 'Alias') {
          // Alias wraps a ColumnRef (Phase 1) — extract the value, key by the alias name
          if (item.expr.type !== 'ColumnRef') {
            throw new Error('project: only ColumnRef aliases supported in Phase 1');
          }
          out[item.alias] = resolvePath(row, item.expr.path);
        } else {
          throw new Error(`project: unsupported projection item type ${item.type}`);
        }
      }
      return out;
    }
    ```

    **DO NOT** filter out null/undefined values from the output — explicit nulls are valid query results, and absent fields show up as `undefined` which `JSON.stringify` correctly drops anyway.

    **Imports:**

    **Function:**
    ```js
    export async function* execute(plan, filePath) {
      // Phase 1 guards
      if (plan.groupBy !== null) {
        throw new Error('GROUP BY is not supported in Phase 1 (lands in Phase 2)');
      }
      if (plan.sort !== null) {
        throw new Error('ORDER BY is not supported in Phase 1 (lands in Phase 2)');
      }
      if (plan.limit !== null) {
        throw new Error('LIMIT/OFFSET are not supported in Phase 1 (lands in Phase 2)');
      }

      const rl = createInterface({
        input: createReadStream(filePath, { encoding: 'utf8' }),
        crlfDelay: Infinity,
      });

      let lineNo = 0;
      for await (const rawLine of rl) {
        lineNo++;
        const line = rawLine.trim();
        if (line === '') continue;          // blank lines are not malformed

        let row;
        try {
          row = JSON.parse(line);
        } catch (err) {
          process.stderr.write(`whoodb: skipping malformed JSON line ${lineNo}: ${err.message}\n`);
          continue;
        }

        if (!evalWhere(plan.filter, row)) continue;
        yield project(plan.projection, row);
      }
    }
    ```

    **Notes embedded in code:**
    - `crlfDelay: Infinity` — handle Windows line endings transparently (research/stack.md mentions this).
    - Encoding `utf8` is explicit to dodge default-encoding portability issues.
    - The malformed-line warning format `whoodb: skipping malformed JSON line N: <message>` is the locked format; it appears in tests in Task 3 via stderr capture. The substring `'malformed'` MUST appear so the SRC-05 success-criterion test can match on it.
    - The function is an `async function*` (async generator) so callers can do `for await (const out of execute(plan, file))`.
    - DO NOT yield empty objects, DO NOT yield `undefined`. If WHERE returns false, just `continue`.
    - DO NOT keep any in-memory accumulator — the WHERE-only path is O(1) memory by design (FILT-07).
